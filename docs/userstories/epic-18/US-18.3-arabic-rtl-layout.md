## User Story

**As an** Arabic-speaking guest,
**I want** the Destinity vButler app to display in a right-to-left layout when I select Arabic,
**So that** the interface feels natural and readable in my native language direction, consistent with the Arabic writing system.

---

## Background / Context

Arabic is written and read right-to-left, meaning that text, layout flow, and navigation conventions are all mirrored compared to LTR languages. For Arabic-speaking guests — a significant segment of Browns Hotels & Resorts international visitors from the Middle East — a proper RTL implementation is the difference between a professional, welcoming product and one that feels broken or inaccessible. This story covers the comprehensive RTL implementation across every UI component in the app.

---

## Acceptance Criteria

- [ ] When Arabic is selected, `document.documentElement.dir` is set to `"rtl"` and `document.documentElement.lang` is set to `"ar"`.
- [ ] When Arabic is selected, the `rtl` CSS class is added to `<body>` to enable Tailwind RTL utility overrides.
- [ ] When any non-Arabic language is selected, `document.documentElement.dir` reverts to `"ltr"`, `lang` updates to the new code, and the `rtl` class is removed from `<body>`.
- [ ] All flexbox row layouts reverse direction: elements that flow left-to-right in LTR flow right-to-left in RTL (using `rtl:flex-row-reverse` or CSS logical properties).
- [ ] All horizontal margins and paddings swap sides: `ml-*` / `mr-*` swap, `pl-*` / `pr-*` swap (using Tailwind RTL variants: `rtl:ml-*`, `rtl:mr-*`, `rtl:pl-*`, `rtl:pr-*`, or CSS `margin-inline-start/end`).
- [ ] All text aligns right by default in RTL mode (inherited from `dir="rtl"` on `<html>`); explicit `text-left` must be replaced with `ltr:text-left rtl:text-right` or removed.
- [ ] The bottom navigation tab bar reverses its item order in RTL so that the first tab (Home/Dashboard) appears on the right side.
- [ ] Directional icons — back arrows, chevrons, forward arrows — are horizontally mirrored in RTL using `rtl:scale-x-[-1]` (CSS transform `scaleX(-1)`).
- [ ] Non-directional icons (bell, star, heart, settings gear, trash) are NOT mirrored.
- [ ] All input fields align text to the right with the cursor starting at the right edge in RTL.
- [ ] **All** pages and components are verified for RTL correctness: auth screens, dashboard, reservation cards, service request screens, dining menus, wellness/spa, housekeeping, transport, wake-up call, experiences, local guide, billing, complaints, profile, settings, notifications, modals, drawers, toasts.
- [ ] The RTL layout applies immediately on language selection without a full page reload.
- [ ] The app correctly reverts to LTR when switching away from Arabic.

## UI / UX Notes

- Arabic font: Use a system font stack that includes Arabic-capable fonts: `font-family: 'Segoe UI', 'Helvetica Neue', Arial, 'Noto Sans Arabic', sans-serif`. For production: consider loading 'Noto Sans Arabic' from Google Fonts only when Arabic is active.
- Line height: Arabic script has taller characters; increase `line-height` to `1.8` for body text when RTL is active (add `rtl:leading-relaxed` or similar).
- Number formatting in Arabic: keep numerals as Western Arabic (0-9) rather than Eastern Arabic-Indic (٠-٩) — hotel guests in the GCC are accustomed to Western numerals in apps.
- LKR currency symbol: always LTR regardless of layout direction (wrap in `<span dir="ltr">LKR 2,500</span>`).
- Toast notifications: text direction flips to right-aligned; icon moves to the right side of the toast.
- Modal close button (X): moves from top-right to top-left in RTL (or use `end-*` Tailwind classes which auto-swap).
- Forms: labels above fields (not inline labels) work better in RTL and avoid directional ambiguity.
- Phone number inputs and numerical fields: use `dir="ltr"` on those specific inputs even in RTL mode (phone numbers, passport numbers, credit card numbers are always LTR).
- Swipe gestures: on the notification screen, swipe-to-delete reverses direction (swipe right in RTL to reveal the delete action).
- Bottom nav: tab order visually reverses; the "active" indicator (brand underline/pip) still aligns correctly under the active tab.

## Technical Notes

- Tailwind RTL plugin: Ensure `tailwind.config.js` includes `plugins: [require('tailwindcss-rtl')]` OR use Tailwind v3's built-in `rtl:` variant (enabled by default when `dir="rtl"` is on `<html>`).
- Tailwind v3 RTL support: The `rtl:` prefix works natively with `<html dir="rtl">`. No plugin needed for Tailwind v3+.
- CSS logical properties alternative: Instead of `ml-4 rtl:ml-0 rtl:mr-4`, use `ms-4` (margin-inline-start) which automatically adapts to the document direction. Preferred for new components.
- `I18nService.setLanguage('ar')` already sets `dir` and `body.classList` (see US-18.1). No additional code needed in individual components.
- Directional icon mirror pattern:
  ```html
  <!-- Back arrow: mirrors in RTL -->
  <svg class="w-6 h-6 rtl:scale-x-[-1]" ...><!-- arrow-left path --></svg>

  <!-- Bell icon: does NOT mirror -->
  <svg class="w-6 h-6" ...><!-- bell path --></svg>
  ```
- Bottom nav RTL:
  ```html
  <nav class="flex rtl:flex-row-reverse">
    <a href="#/dashboard">...</a>
    <a href="#/services">...</a>
    <!-- etc. -->
  </nav>
  ```
- LTR overrides for specific inputs:
  ```html
  <input type="tel" dir="ltr" class="text-left" placeholder="+94 77 000 0000">
  <input type="text" dir="ltr" class="text-left" placeholder="Passport number">
  ```
- RTL-aware absolute positioning: use `start-*` and `end-*` Tailwind utilities instead of `left-*` / `right-*` for elements that must flip (e.g., notification badge, dropdown arrows, input icons).
  - Example: `start-0` = `left-0` in LTR, `right-0` in RTL.
- Testing checklist matrix: create a spreadsheet tracking each screen × [layout flip, icon mirror, text align, input direction, badge position] as pass/fail cells.

## Test Cases

### Happy Path

1. Open app in English — verify `<html dir="ltr">`.
2. Switch language to Arabic.
3. Verify `<html dir="rtl" lang="ar">` and `<body class="... rtl">`.
4. Verify dashboard layout is mirrored: greeting text on right, avatar on left (if header has avatar-right in LTR, it moves left in RTL — or vice versa per design).
5. Verify bottom nav tab order is reversed (Dashboard tab on far right in RTL).
6. Verify the back arrow icon on the Profile screen is mirrored (points right, meaning "go forward" visually in Arabic convention = go back in LTR = go forward in RTL reading direction).
7. Verify bell icon in header is NOT mirrored.
8. Verify a toast notification is right-aligned with the icon on the right.
9. Switch back to English — verify `dir="ltr"` restored; all layout reverts to LTR.

### Component-Level Checks (all must pass)

| Component | Check |
|-----------|-------|
| Auth screens (login/register) | Form fields right-aligned, labels right, button full-width |
| Dashboard cards | Card content flows right-to-left |
| Service request form | Labels above fields, text right-aligned |
| Billing table | Column order reversed; LKR amounts remain LTR |
| Notification items | Icon on right, text left of icon, timestamp on left |
| Modal dialogs | Close (X) button top-left; content right-aligned |
| Bottom drawer | Handle centred; option text right-aligned; icon left of text |
| Profile chips | Chips flow from right to left |
| Date picker | Calendar grid has Sunday on right for Arabic locales |
| Input with prefix icon | Prefix icon on right; text starts from right |

### Edge Cases

- **Switch Arabic → German → Arabic:** Each switch applies correct dir and body class; no stale class from intermediate state.
- **Persistent on reload:** `vb:language = 'ar'` causes boot to set RTL immediately before Alpine renders; no LTR flash.
- **Nested flex containers:** All layers reverse; no child container remains LTR if parent is RTL (CSS inheritance handles this).
- **Third-party components (e.g., Choices.js, Day.js):** Choices.js renders correctly RTL when parent is `dir="rtl"`. Day.js locales: set `dayjs.locale('ar')` when Arabic is active for RTL-formatted date strings if needed.
- **150% browser zoom in RTL:** Layout remains usable; no overflow or truncation.
- **LTR-locked inputs in RTL page:** Passport number and phone number inputs remain LTR text direction; verified by entering text and checking cursor position.
