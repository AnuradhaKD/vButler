## User Story

**As a** guest with accessibility needs,
**I want** the Destinity vButler app to be fully usable with assistive technologies and at increased font sizes,
**So that** I can independently manage my stay at Browns Hotels & Resorts without needing assistance from staff due to app usability barriers.

---

## Background / Context

Browns Hotels & Resorts serves guests of all abilities, including those who use screen readers, keyboard navigation, or increased browser font sizes due to visual impairments. WCAG 2.1 AA compliance is both an ethical obligation and, in many jurisdictions, a legal requirement for digital products. A Lighthouse accessibility score of ≥ 90 is the minimum measurable threshold, but the goal is a genuinely usable product for guests with visual, motor, or cognitive accessibility needs.

## Acceptance Criteria

### Colour Contrast (WCAG 2.1 SC 1.4.3 / 1.4.11)

- [ ] All normal body text (< 18px or < 14px bold) meets a minimum contrast ratio of 4.5:1 against its background.
- [ ] All large text (≥ 18px or ≥ 14px bold) meets a minimum contrast ratio of 3:1 against its background.
- [ ] The brand primary colour `#003c52` on white `#ffffff` is verified to meet ≥ 4.5:1 (verified ratio: approximately 9.7:1 — passes).
- [ ] All status badge colour combinations are verified: e.g., white text on green (#16A34A) for "Completed", white text on amber (#D97706) for "Pending", white text on red (#DC2626) for "Cancelled".
- [ ] All muted/secondary text colours are verified against their backgrounds in both light and dark mode.
- [ ] Dark mode: all text colours re-verified for contrast on dark surface backgrounds.

### Touch Targets (WCAG 2.1 SC 2.5.5)

- [ ] All interactive elements have a minimum touch target size of 44×44px (measured as the tappable area, not the visual icon size).
- [ ] Bottom navigation tabs have a minimum height of 48px.
- [ ] Star rating inputs (if present on feedback screens) have a minimum touch target of 44×44px per star, with sufficient spacing.
- [ ] Icon buttons (bell, close X, back arrow, eye toggle, etc.) use padding to reach 44×44px even when the icon itself is 20–24px.

### ARIA Roles and Labels (WCAG 2.1 SC 1.3.1, 4.1.2)

- [ ] All icon-only buttons have a descriptive `aria-label` (e.g., `<button aria-label="Close">`, `<button aria-label="Notifications">`).
- [ ] All form input fields have an associated `<label>` element linked via `for`/`id`, or `aria-label`/`aria-labelledby`.
- [ ] Inline validation error messages use `role="alert"` so screen readers announce them immediately on appearance.
- [ ] The Do Not Disturb toggle uses `aria-live="polite"` on the status text so screen readers announce the new state ("Do Not Disturb: On") when toggled.
- [ ] All modal dialogs use `role="dialog"` and `aria-modal="true"` and include an `aria-labelledby` pointing to the modal's heading.
- [ ] Bottom navigation uses `role="tablist"` with each tab having `role="tab"` and `aria-selected="true/false"`.
- [ ] Loading states use `aria-busy="true"` on the relevant container; skeleton screens include `aria-label="Loading content"`.
- [ ] Progress indicators (e.g., pre-arrival checklist progress, multi-step forms) use `aria-valuenow`, `aria-valuemin`, `aria-valuemax` on the progress bar element.
- [ ] Expandable/collapsible sections (profile cards, FAQ items) use `aria-expanded="true/false"` on the trigger button and `aria-controls` referencing the panel ID.

### Keyboard Navigation (WCAG 2.1 SC 2.1.1, 2.4.3, 2.4.7)

- [ ] All interactive elements are reachable via the Tab key in a logical left-to-right, top-to-bottom order (or right-to-left, top-to-bottom in RTL).
- [ ] No keyboard focus traps exist outside of modal dialogs (modals intentionally trap focus while open).
- [ ] All modal dialogs trap focus within the modal and release it on close, returning focus to the trigger element.
- [ ] All modals close on pressing the Escape key.
- [ ] All dropdown menus are operable with keyboard: Tab/Shift+Tab to navigate options, Enter to select, Escape to close.
- [ ] A visible focus ring is present on all focused elements: `focus:ring-2 focus:ring-brand focus:ring-offset-2` (Tailwind); the ring must have sufficient contrast.
- [ ] The focus ring is visible in both light and dark mode.
- [ ] No elements use `tabindex` values greater than 0 (avoid positive tabindex).
- [ ] Skip-to-content link: a visually hidden "Skip to main content" link is the first focusable element on every page; it becomes visible on Tab focus.

### Font Scaling (WCAG 2.1 SC 1.4.4)

- [ ] The app is fully usable at 150% browser zoom (Ctrl/Cmd + "+" three times); no content is clipped, overlapping, or requires horizontal scrolling.
- [ ] The app is fully usable at 200% browser zoom; layout may reflow to single-column but all content remains accessible.
- [ ] All font sizes use `rem` or `em` units, not fixed `px`, to respect user browser font-size preferences.
- [ ] All spacing (padding, margin, gap) uses `rem`-based Tailwind classes (which use `rem` by default) to scale with font size.
- [ ] No fixed-height containers clip text when font size is increased.

## UI / UX Notes

- Focus ring: `focus-visible:ring-2 focus-visible:ring-offset-2` using `:focus-visible` (not `:focus`) to avoid showing the ring on mouse/touch interactions while still showing it on keyboard focus.
- Focus ring colour: in light mode, brand #003c52; in dark mode, a lighter brand tint or white to contrast against dark backgrounds.
- "Skip to main content" link: `position: absolute; top: -100%; left: 0;` — moves into view when focused: `focus:top-0`.
- Error messages: `role="alert"` causes screen readers to announce immediately; use sparingly — only for actual errors, not informational messages.
- `aria-live="polite"` vs `aria-live="assertive"`: use `polite` for status updates (DND toggle, form save success); use `assertive` only for critical errors.
- Tooltips on icon buttons: `title` attribute alone is insufficient (not announced on mobile); always pair with `aria-label`.
- Colour alone: never use colour as the only means of conveying information (e.g., status badges should include a text label, not just a coloured dot).

## Technical Notes

- **axe DevTools audit:** Install the axe DevTools browser extension; run on every unique screen template and fix all Critical and Serious issues before marking the story done.
- **Lighthouse:** Run `Lighthouse > Accessibility` in Chrome DevTools on: login, dashboard, services, dining, billing, complaints, profile, notifications screens. Target score ≥ 90 on each.
- **Tailwind focus utilities:** Add to `tailwind.config.js`:
  ```js
  theme: {
    extend: {
      ringColor: {
        brand: '#003c52',
      }
    }
  }
  ```
  Apply: `focus-visible:ring-2 focus-visible:ring-brand focus-visible:ring-offset-2`.
- **Modal focus trap implementation:**
  ```js
  function trapFocus(modal) {
    const focusable = modal.querySelectorAll(
      'a[href], button:not([disabled]), input, select, textarea, [tabindex]:not([tabindex="-1"])'
    );
    const first = focusable[0];
    const last = focusable[focusable.length - 1];
    modal.addEventListener('keydown', e => {
      if (e.key !== 'Tab') return;
      if (e.shiftKey) {
        if (document.activeElement === first) { e.preventDefault(); last.focus(); }
      } else {
        if (document.activeElement === last) { e.preventDefault(); first.focus(); }
      }
    });
    first.focus();
  }
  ```
- **Contrast checking tool:** [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/) or `axe` programmatic API. Run checks for: brand on white, muted text on light bg, white text on brand, all badge colour combinations.
- **Contrast values to document and verify:**

  | Foreground | Background | Ratio | Pass/Fail (AA Normal) |
  |------------|------------|-------|-----------------------|
  | #003c52 (brand) | #FFFFFF (white) | ~9.7:1 | Pass |
  | #FFFFFF (white) | #003c52 (brand) | ~9.7:1 | Pass |
  | #6B7280 (gray-500 muted) | #FFFFFF (white) | ~4.6:1 | Pass |
  | #FFFFFF (white) | #16A34A (green-600) | ~4.5:1 | Pass |
  | #FFFFFF (white) | #D97706 (amber-600) | ~3.0:1 | Fail — use darker shade #B45309 |
  | #FFFFFF (white) | #DC2626 (red-600) | ~4.7:1 | Pass |
  | #1F2937 (gray-800) | #F9FAFB (gray-50) | ~16:1 | Pass |

- Amber status badges: switch to `#B45309` (amber-700) background to meet 4.5:1 with white text.
- `rem` units: Tailwind's default configuration uses `rem`; verify no raw `px` values are used for `font-size` in custom CSS.

## Test Cases

### Happy Path

1. Open Chrome DevTools → Lighthouse → Accessibility audit on `/index.html#/dashboard`.
2. Verify score ≥ 90.
3. Tab through the dashboard — verify every interactive element receives a visible focus ring in sequence.
4. Open a modal — verify Tab cycles within the modal only; Escape closes the modal.
5. Activate a screen reader (NVDA/JAWS on Windows or VoiceOver on macOS/iOS).
6. Navigate to the login form — verify field labels are announced correctly on focus.
7. Submit the form with an empty email — verify `role="alert"` error is announced immediately.
8. Open the notification bell dropdown — verify `aria-expanded` changes are announced.
9. Set browser zoom to 150% — verify all content visible without horizontal scroll on dashboard.
10. Set browser zoom to 200% — verify all content reflows to single column; no text is cut off.

### Edge Cases

- **High Contrast Mode (Windows):** App renders usably; custom colours may be overridden by OS; critical information still conveyed via text/shape, not colour alone.
- **iOS VoiceOver swipe navigation:** All interactive elements discoverable via swipe-right; no invisible tappable areas.
- **NVDA + Firefox:** Test at least the login, dashboard, and billing screens with NVDA in browse mode.
- **Browser zoom 150% in RTL (Arabic):** No overlap or cutoff; all Arabic text scales correctly.
- **Amber badge after colour fix:** Verify axe DevTools reports 0 contrast errors for the `#B45309` badge.
