## User Story

**As a** developer building Destinity vButler,
**I want** a complete, well-structured set of translation JSON files for all five supported languages,
**So that** `I18nService` has the correct key structure to serve any translated string in the app without key-resolution errors.

---

## Background / Context

The translation files are the backbone of the internationalisation system. Without a correctly structured `en.json` as the master reference, other language files will have key mismatches that cause runtime fallback chains or missing UI strings. Phase 1 prioritises a fully complete English file with every UI string in the app, while the other four language files have identical key structures with English placeholder values — ready for professional translation as specified in Browns Hotels & Resorts PRD §A5.

---

## Acceptance Criteria

- [ ] Five translation JSON files exist at `/data/i18n/`: `en.json`, `si.json`, `ta.json`, `ar.json`, `de.json`.
- [ ] `en.json` contains **all** translation keys used in every template across every screen, organised into the namespace hierarchy defined below.
- [ ] `si.json`, `ta.json`, `ar.json`, and `de.json` have **identical key structures** to `en.json` — no missing keys, no extra keys.
- [ ] Phase 1: `si.json`, `ta.json`, `ar.json`, and `de.json` use English placeholder values for all strings (with a `[SI]`, `[TA]`, `[AR]`, `[DE]` prefix on each value to make placeholders visually identifiable during development).
- [ ] The `common` namespace covers all reusable action strings used across multiple screens.
- [ ] Template variable placeholders use `{{variableName}}` syntax in all language files.
- [ ] A build/lint script or npm script `npm run i18n:validate` compares all five files against `en.json` and reports any missing or extra keys.
- [ ] All JSON files are valid, parseable JSON (no trailing commas, no comments).

- Validation script (`/scripts/validate-i18n.js`):
  ```js
  const en = require('../data/i18n/en.json');
  const langs = ['si', 'ta', 'ar', 'de'];

  function getKeys(obj, prefix = '') {
    return Object.entries(obj).flatMap(([k, v]) =>
      typeof v === 'object' && !Array.isArray(v)
        ? getKeys(v, `${prefix}${k}.`)
        : [`${prefix}${k}`]
    );
  }
  const enKeys = new Set(getKeys(en));
  langs.forEach(code => {
    const lang = require(`../data/i18n/${code}.json`);
    const langKeys = new Set(getKeys(lang));
    const missing = [...enKeys].filter(k => !langKeys.has(k));
    const extra   = [...langKeys].filter(k => !enKeys.has(k));
    if (missing.length) console.error(`${code}: MISSING keys:`, missing);
    if (extra.length)   console.warn(`${code}: EXTRA keys:`, extra);
    if (!missing.length && !extra.length) console.log(`${code}: ✓ OK`);
  });
  ```

## Test Cases

### Happy Path

1. Run `npm run i18n:validate` — verify output shows `si: ✓ OK`, `ta: ✓ OK`, `ar: ✓ OK`, `de: ✓ OK`.
2. Select each language in turn and scan every screen — verify no raw key strings (e.g., `auth.login.title`) are visible; all strings show either translated text or `[XX] English placeholder`.
3. In `en.json`, verify the `notifications.templates.billingCharge.body` contains `{{amount}}` and `{{description}}` placeholders.
4. Call `I18nService.t('notifications.templates.billingCharge.body', { amount: '2,500', description: 'minibar' })` — verify output: "LKR 2,500 has been added to your bill for minibar."

### Edge Cases

- **Add a new key to `en.json` but forget to add it to `ar.json`:** Validation script reports `ar: MISSING keys: [new.key]`; CI build fails.
- **Add an extra key to `de.json` not in `en.json`:** Validation script reports `de: EXTRA keys: [de.extraKey]`; developer corrects.
- **JSON syntax error in `ta.json`:** `JSON.parse` throws on load; `I18nService.load('ta')` catches and falls back to English; validation script also detects parse error.
- **Deeply nested key (5 levels):** `getKeys()` resolver handles without stack overflow.
- **Empty string value for a key in `si.json`:** Allowed (empty string is a valid placeholder); `t()` returns empty string; not treated as missing.
