## User Story

**As a** hotel guest,
**I want** to permanently delete my account and all associated data from the app,
**So that** I have full control over my personal information and can exercise my right to erasure.

---

## Background / Context

Browns Hotels & Resorts is committed to responsible data handling and GDPR-aligned principles. Guests should be able to request full data erasure at any time. Because Destinity vButler is a client-side app in Phase 1, all data resides in the guest's own browser localStorage; account deletion means clearing all `vb:*` keys from that storage. A deliberate two-step confirmation with explicit text input prevents accidental deletion of stay history, preferences, and billing records.

---

## Acceptance Criteria

- [ ] The "Delete Account" option is visible in Settings → Account section and uses destructive red text to signal danger.
- [ ] Tapping "Delete Account" opens a two-step confirmation modal.
- [ ] **Step 1 of the modal** displays the message: "Are you sure? This will permanently delete your account and all data. This action cannot be undone." with two buttons: [Cancel] and [Continue].
- [ ] Tapping [Cancel] in Step 1 dismisses the modal with no changes.
- [ ] Tapping [Continue] in Step 1 advances to Step 2 within the same modal (no new screen).
- [ ] **Step 2 of the modal** displays the instruction: "To confirm, type DELETE in the box below." with a text input and two buttons: [Cancel] and [Delete Account].
- [ ] The [Delete Account] button in Step 2 is disabled until the text input contains exactly "DELETE" (case-sensitive).
- [ ] Tapping [Cancel] in Step 2 dismisses the modal with no changes.
- [ ] On confirmed deletion (correct "DELETE" input + [Delete Account] tap):
  - All `vb:*` keys are removed from `localStorage` using: `Object.keys(localStorage).filter(k => k.startsWith('vb:')).forEach(k => localStorage.removeItem(k))`.
  - The app redirects to `#/login`.
  - A toast notification reads: "Your account has been deleted."
- [ ] The deletion removes at minimum: `vb:session`, `vb:profile`, `vb:auth`, `vb:reservations`, `vb:requests`, `vb:complaints`, `vb:notifications`, `vb:wishlist`, `vb:theme`, `vb:language`, `vb:billing`, and any other `vb:` prefixed keys.
- [ ] After deletion and redirect to `#/login`, the login screen is fully fresh — no pre-filled email, no session, no prior state.
- [ ] The modal is accessible: `role="dialog"`, `aria-modal="true"`, focus is trapped within the modal while open.
- [ ] Pressing the Escape key dismisses the modal (returns to Step 1 if in Step 2, or closes entirely if in Step 1).

## UI / UX Notes

- Modal: centred overlay on desktop; bottom-sheet style on mobile (max-height 60vh, scrollable content).
- Backdrop: semi-transparent dark (rgba(0,0,0,0.5)); tapping backdrop in Step 1 closes the modal; tapping backdrop in Step 2 does NOT close (prevents accidental dismissal mid-flow).
- Step 1 icon: large warning/alert triangle icon (amber/orange) centred above the heading.
- Step 1 heading: "Delete Account?" — 18px bold.
- Step 1 body text: 14px, muted; max-width 320px, centred.
- [Cancel] button: secondary/ghost style.
- [Continue] button: amber/warning style (not yet the full destructive red — this is the warning step).
- Step 2 icon: large trash/delete icon in red.
- Step 2 heading: "This cannot be undone." — 18px bold, red.
- Step 2 text input: placeholder text "Type DELETE to confirm"; styled with a red border when incorrect text is entered and user has interacted; no border colour change until after first keystroke.
- [Delete Account] button in Step 2: full destructive red (#DC2626) background; white text; disabled state: opacity-40, cursor-not-allowed.
- Success toast: bottom-centre, red/destructive variant, "Your account has been deleted.", auto-dismiss 5 seconds.
- Transition between Step 1 and Step 2: fade-crossfade or slide-left animation within the modal container.

## Technical Notes

- Alpine.js component `x-data="accountDeletion()"` with: `step: 1`, `confirmText: ''`, `isOpen: false`.
- Computed: `canDelete` = `this.confirmText === 'DELETE'`.
- Deletion function:
  ```js
  deleteAccount() {
    Object.keys(localStorage)
      .filter(k => k.startsWith('vb:'))
      .forEach(k => localStorage.removeItem(k));
    Alpine.store('session').clear(); // reset all Alpine stores
    NotificationService.create({ type: 'info', title: 'Account Deleted', body: 'Your account has been deleted.' }); // will not persist as storage is cleared
    window.location.hash = '#/login';
    // Toast shown via a URL query param or sessionStorage flag read by login screen
  }
  ```
- Because all localStorage is cleared before the toast can display via Alpine, use `sessionStorage.setItem('vb:deleted', 'true')` to pass the toast message to the login screen, then clear it after displaying.
- Alpine stores to reset on deletion: `profile`, `session`, `notifications`, `ui` — call a `reset()` method on each store.
- Focus trap: on modal open, `document.getElementById('modal').focus()` and intercept Tab/Shift+Tab to cycle within modal elements only.
- Escape key: `@keydown.escape.window="closeModal()"`.
- The "DELETE" check must be exact string match — not trimmed, not case-insensitive: `confirmText === 'DELETE'`.

## Test Cases

### Happy Path

1. Navigate to Settings → Account → "Delete Account".
2. Verify Step 1 modal appears with warning icon, descriptive text, [Cancel] and [Continue].
3. Tap [Continue] — verify modal advances to Step 2 with trash icon, red heading, and text input.
4. Verify [Delete Account] button is disabled (input is empty).
5. Type "delete" (lowercase) — verify button remains disabled (case-sensitive check).
6. Clear and type "DELETE" — verify [Delete Account] button becomes active (red, enabled).
7. Tap [Delete Account].
8. Verify the app redirects to `#/login`.
9. Verify toast "Your account has been deleted." appears on the login screen.
10. Open browser DevTools → Application → localStorage — verify no `vb:*` keys remain.
11. Attempt to navigate to `#/dashboard` — verify router redirects back to `#/login`.

### Edge Cases

- **Cancel in Step 1:** Modal closes; all localStorage keys intact; user remains on Settings screen.
- **Cancel in Step 2:** Modal closes; all localStorage keys intact; user remains on Settings screen.
- **Backdrop tap in Step 2:** Modal does NOT close (prevents accidental dismissal).
- **Escape in Step 1:** Modal closes.
- **Escape in Step 2:** Modal returns to Step 1 (or closes — team decision; document consistently).
- **"delete" (lowercase):** [Delete Account] button disabled; no deletion occurs.
- **"DELETE " (trailing space):** Button disabled; exact match required.
- **Multiple `vb:` keys added by future epics:** Deletion loop is generic (`filter(k => k.startsWith('vb:'))`) so all future keys are automatically cleared.
- **Dark mode:** Modal, buttons, text input, and destructive red elements all render correctly.
- **RTL (Arabic):** Modal layout mirrors correctly; button order reverses; text input is LTR (since "DELETE" is Latin).
