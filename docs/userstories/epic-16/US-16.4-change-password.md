## User Story

**As a** hotel guest,
**I want** to change my account password from within the app,
**So that** I can keep my Destinity vButler account secure if I suspect it has been compromised or simply want to update my credentials.

---

## Background / Context

Security is a key concern for guests who store personal information such as passport numbers, room preferences, and billing details in Destinity vButler. Allowing guests to change their password directly in the app — without needing to contact reception or use a forgotten-password flow — reduces friction and builds trust. The password change must use the same hashing mechanism as the initial registration to ensure consistency.

---

## Acceptance Criteria

- [ ] The Change Password screen or modal is accessible from Settings → Account → "Change Password".
- [ ] The form contains three fields: "Current Password", "New Password", and "Confirm New Password".
- [ ] Each field has a show/hide toggle (eye icon) to reveal or mask the password characters.
- [ ] "Current Password" is validated against the stored hash in `vb:auth.passwordHash` using SubtleCrypto SHA-256 before any changes are made.
- [ ] If the current password does not match, an inline error is shown directly below the "Current Password" field: "Incorrect current password."
- [ ] "New Password" must be at least 8 characters; if not, an inline error reads: "Password must be at least 8 characters."
- [ ] "Confirm New Password" must exactly match "New Password"; if not, an inline error reads: "Passwords do not match."
- [ ] The submit button is labelled "Update Password" and is disabled until all three fields are non-empty and no validation errors are present.
- [ ] On successful validation and hash match, the new password is hashed with SHA-256 via SubtleCrypto and the resulting hash replaces the value in `vb:auth.passwordHash`.
- [ ] On success, a toast notification reads "Password updated successfully" and the form/modal is dismissed.
- [ ] All three fields are cleared after a successful change.
- [ ] The form renders correctly in both light and dark mode and in RTL layout.

## UI / UX Notes

- Presentation: can be implemented as a dedicated screen (`#/settings/change-password`) or a full-screen modal slide-up; modal preferred for mobile continuity.
- Header: "Change Password" title with a back/close button (X icon top-right or back arrow top-left).
- Field layout: vertical stack, 16px gap; each field is a labelled input group.
- Show/hide toggle: eye / eye-slash icon positioned inside the input on the right; 44×44px tap target.
- Inline error text: 12px, red (#EF4444 or theme error colour), appears with a subtle fade-in below the affected field.
- "Update Password" button: full-width, brand primary, disabled state uses reduced opacity (opacity-50) and no pointer cursor.
- Loading state: button shows a spinner and "Updating…" label while the async SubtleCrypto hash operation runs.
- Success toast: bottom-centre, green variant, "Password updated successfully", auto-dismiss 3 seconds.
- Password strength indicator (optional enhancement): thin coloured bar below "New Password" field — red (< 8 chars), orange (8–11 chars, letters only), green (12+ chars or mixed).

## Technical Notes

- Hashing function: reuse the same utility as registration — `async function hashPassword(plain) { const buf = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(plain)); return Array.from(new Uint8Array(buf)).map(b => b.toString(16).padStart(2, '0')).join(''); }`.
- Verification: `const currentHash = await hashPassword(currentPasswordInput); if (currentHash !== StorageService.get('vb:auth').passwordHash) { showError('current', 'Incorrect current password.'); return; }`.
- Write: `const newHash = await hashPassword(newPasswordInput); const auth = StorageService.get('vb:auth'); auth.passwordHash = newHash; StorageService.set('vb:auth', auth);`.
- Do NOT store plaintext passwords at any point.
- Alpine component: `x-data="changePasswordForm()"` with `currentPassword`, `newPassword`, `confirmPassword` fields; `showCurrent`, `showNew`, `showConfirm` booleans for visibility toggles.
- Error state: `errors.current`, `errors.new`, `errors.confirm` reactive properties.
- The SubtleCrypto API is async; wrap the submit handler in an `async` function and use `await`; disable the button and show a spinner during processing.
- Route guard: this screen/modal must only be accessible to authenticated sessions (valid `vb:session`).

## Test Cases

### Happy Path

1. Navigate to Settings → Account → "Change Password".
2. Enter the correct current password.
3. Enter a new password "MyNewPass2025" (13 chars).
4. Enter "MyNewPass2025" in Confirm field.
5. Tap "Update Password".
6. Verify "Password updated successfully" toast appears.
7. Log out and attempt to log in with the old password — verify login fails.
8. Log in with the new password "MyNewPass2025" — verify login succeeds.

### Edge Cases

- **Wrong current password:** Enter "WrongPass123" as current password → inline error "Incorrect current password" appears; `vb:auth.passwordHash` unchanged.
- **New password = 7 characters:** Inline error "Password must be at least 8 characters"; submit button remains disabled.
- **Passwords don't match:** "NewPass123" vs "NewPass456" → inline error "Passwords do not match"; submit disabled.
- **New password same as current:** No explicit restriction in scope; password is updated (hash will be same value — effectively a no-op but no error).
- **Show/hide toggle:** Toggle each field independently; verify characters show/hide correctly without affecting the other fields.
- **Form submitted empty:** All three fields show required errors; no network/storage call made.
- **Dark mode:** All fields, error states, and button render correctly.
- **Arabic RTL:** Field labels right-aligned; show/hide icons on the left side of inputs; error text right-aligned.
- **Multiple rapid taps on submit:** Button disabled after first tap until async operation resolves; only one hash comparison and one storage write occur.
