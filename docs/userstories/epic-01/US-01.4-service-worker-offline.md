## User Story

**As a** guest on poor hotel Wi-Fi or with no connectivity,
**I want** the app to still load and display my reservation information,
**So that** I can access my booking details and navigate the app even when the network is unavailable.

---

## Background / Context

Hotel guests frequently experience unreliable Wi-Fi, especially in rooms distant from access points or during peak usage times. A PWA service worker allows Destinity vButler to pre-cache the application shell and data files so the app loads reliably regardless of connectivity. For a guest trying to check their check-out time or review their bill at 2am with weak signal, offline capability is not optional — it directly impacts the quality of the stay experience.

---

## Acceptance Criteria

- [ ] A `sw.js` file exists at the project root (same level as `index.html`)
- [ ] `index.html` registers the service worker on first load via `navigator.serviceWorker.register('/sw.js')` inside a feature-detection check
- [ ] The service worker precaches the app shell on `install` event: `index.html`, all linked CSS, all `js/` modules, icon assets, and font files (if self-hosted)
- [ ] App shell uses a **cache-first** strategy: serve from cache immediately, no network request for shell assets
- [ ] JSON data files (`/data/mock/*.json`) use a **network-first with cache fallback** strategy: try network first; if offline, serve from cache
- [ ] Property images are cached on **first load** (as they are requested by the page) using a cache-first strategy for subsequent loads
- [ ] When the user is offline and requests a data file not yet cached, the relevant UI section displays a graceful "You're offline" empty state (using the shared `EmptyState` component from US-01.7) rather than a JavaScript error
- [ ] Service worker updates silently in the background: new SW calls `self.skipWaiting()` in `install` event and `clients.claim()` in `activate` event
- [ ] Old cache versions are purged in the `activate` event (cache versioning via named cache string constant e.g., `CACHE_V1`)
- [ ] Service worker does not intercept or cache requests to external CDN domains in a way that breaks font/icon loading
- [ ] Chrome DevTools → Application → Service Workers panel shows the SW as "activated and running"
- [ ] Chrome DevTools → Application → Cache Storage shows cached shell assets after first load

---

## UI / UX Notes

- The "You're offline" state should be non-alarming and informative. Use the `EmptyState` component with a `wifi_off` Material Symbol icon, title "You're offline", and subtitle "Some content isn't available without a connection. We'll reload it when you're back online."
- The offline state should only appear for data-dependent sections, not for the entire app. Navigation, previously loaded reservation cards, and static UI should remain fully functional
- When connectivity is restored, do not force a full page reload — instead, silently re-fetch the failed data and update the UI. Listen to `window.addEventListener('online', ...)` to trigger a refresh of stale content
- A subtle "Offline mode" chip/indicator may appear in the header when the device is offline (optional, P2)

## Technical Notes

- **File:** `sw.js` at project root
- **Cache name versioning:** Use a constant at the top of `sw.js`: `const CACHE_NAME = 'vbutler-v1'`; increment version on each deployment
- **Precache list:** Hardcode the app shell asset list in `sw.js` `PRECACHE_ASSETS` array — include `'/'`, `'/index.html'`, `'/js/router.js'`, `'/js/app.js'`, and all other core JS modules and CSS
- **Cache strategies:**
  - **Cache-first:** App shell assets (`*.html`, `*.js`, `*.css`, icon files) — `caches.match(request).then(cached => cached || fetch(request))`
  - **Network-first:** `/data/mock/*.json` — `fetch(request).then(res => { cache.put(request, res.clone()); return res; }).catch(() => caches.match(request))`
  - **Cache-first (on demand):** Images — intercepted on first request, cached, served from cache subsequently
- **SW registration (in `index.html` or `js/app.js`):**
  ```js
  if ('serviceWorker' in navigator) {
    window.addEventListener('load', () => {
      navigator.serviceWorker.register('/sw.js').catch(console.error);
    });
  }
  ```
- **skipWaiting and clients.claim:**
  ```js
  self.addEventListener('install', e => { e.waitUntil(precache()); self.skipWaiting(); });
  self.addEventListener('activate', e => { e.waitUntil(purgeOldCaches()); self.clients.claim(); });
  ```
- **Offline detection in app:** Use `navigator.onLine` and `window.addEventListener('online'/'offline', ...)` to update an Alpine.js store flag `ui.isOffline`; bind `x-show` on the offline indicator to `$store.ui.isOffline`
- **Important:** External CDN requests (Google Fonts, Tailwind CDN) should use a **network-first** or **stale-while-revalidate** strategy, not cache-first, to avoid serving stale CDN resources

## Test Cases

### Happy Path

1. Load app in Chrome with network enabled — SW installs, caches app shell; verify in DevTools → Cache Storage
2. Set Chrome to offline (DevTools → Network → Offline) — reload app; `index.html` and JS load from cache; app renders
3. Navigate to Reservations while offline — previously loaded reservation data shows from cache; no JavaScript errors
4. Restore network in DevTools → data sections silently re-fetch and update

### Edge Cases

- First-ever load while offline — no cache exists; app cannot load (expected); browser shows default offline error page; this is acceptable and documented
- Service worker update available (new `CACHE_V1` → `CACHE_V2`) — new SW installs and activates without user action; old cache purged; no content disruption
- Data file not yet cached, user goes offline, navigates to that section — "You're offline" EmptyState shown; no thrown errors in console
- Property image not yet cached, user goes offline — `<img>` `onerror` handler shows a brand-colour placeholder rather than a broken image icon
- Service worker registration fails (e.g., served over HTTP in production) — registration silently fails; app still functions online; no console errors thrown to user-visible state
