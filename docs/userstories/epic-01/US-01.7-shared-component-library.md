## User Story

**As a** developer,
**I want** a library of reusable, consistently styled UI components,
**So that** every feature screen uses the same visual patterns for feedback, overlays, status, and empty states without duplicating code.

---

## Background / Context

A shared component library is the foundation of UI consistency. Without it, individual feature developers will create ad-hoc toast messages, custom modals, and status badges that look slightly different from each other, eroding the Browns Hotels brand experience. These components also encode accessibility (ARIA roles, keyboard traps in modals) and animation standards centrally, so feature stories only need to invoke the component — not re-implement the behaviour.

## Acceptance Criteria

### Toast Component
- [ ] Toast supports three variants: `success` (green), `error` (red), `info` (blue)
- [ ] Toast auto-dismisses after 4 seconds
- [ ] Toast slides in from the top (mobile) or top-right (desktop) with a CSS animation
- [ ] Multiple toasts can stack (newest on top, max 3 visible at once)
- [ ] Toast can be manually dismissed by clicking an `×` button
- [ ] `ToastService.show({ message, type })` is callable from any module
- [ ] Toast renders above all other content (z-index layer above modals if triggered simultaneously)

### Modal Component
- [ ] Modal is centred overlay on desktop (max-width 480px, vertically centred)
- [ ] Modal is a bottom drawer on mobile (slides up from bottom, full viewport width, rounded top corners)
- [ ] Backdrop (`bg-black/50`) covers the rest of the screen; clicking backdrop closes the modal
- [ ] Modal traps keyboard focus (Tab cycles within modal only while open)
- [ ] `Escape` key closes the modal
- [ ] Smooth open/close animation: desktop = fade + scale; mobile = slide up/down
- [ ] Modal exposes a `title` slot, `body` slot, and `footer` slot (for action buttons)
- [ ] Alpine.js: modal open/close state managed via `$store.modal` or a dedicated `x-data` component

### StatusBadge Component
- [ ] StatusBadge accepts a `status` prop and renders the correct colour and label:
  - `active` → green background, "Active"
  - `upcoming` → blue background, "Upcoming"
  - `checked_out` → grey background, "Checked Out"
  - `cancelled` → red background, "Cancelled"
  - `submitted` → yellow background, "Submitted"
  - `in_progress` → orange background, "In Progress"
  - `resolved` → green background, "Resolved"
- [ ] Badge uses soft/pastel background with dark text for legibility in both light and dark modes
- [ ] Badge is a small pill shape (rounded-full, px-2.5 py-0.5, text-xs font-medium)

### LoadingState Component
- [ ] Full-screen (or container-relative) spinner overlay with optional message text below spinner
- [ ] Spinner is a CSS-animated ring in brand colour
- [ ] Background overlay is semi-transparent (`bg-white/70 dark:bg-surface-dark/70`)
- [ ] Activated/deactivated via `$store.ui.loading` Alpine store boolean

### EmptyState Component
- [ ] Accepts: icon name (Material Symbol), title string, subtitle string, optional CTA label + action
- [ ] Centred layout with icon (48px, muted colour), bold title, lighter subtitle, optional button
- [ ] Used for: no reservations, offline state, no results, empty list sections

### ReservationContextHeader Component
- [ ] A horizontal strip rendered at the top of all authenticated non-auth pages (below the main app header)
- [ ] Displays: property name (bold), check-in → check-out date range, `StatusBadge` for current reservation status
- [ ] Data sourced from `$store.reservations.active` (the currently selected reservation)
- [ ] Updates reactively when the active reservation changes (via US-03.4 context switcher)
- [ ] Styled: brand colour left border accent or subtle tinted background strip

## UI / UX Notes

- **Toast:** Position `fixed top-4 right-4` on desktop, `fixed top-4 left-1/2 -translate-x-1/2 w-[calc(100%-2rem)]` on mobile. Icon + message + dismiss button. Success: `check_circle` icon; Error: `error` icon; Info: `info` icon
- **Modal / Bottom drawer:** On mobile, the drawer should snap to natural content height (not full screen unless content is very long). Add a drag handle affordance at the top centre. Max height 85vh with internal scroll if needed
- **StatusBadge colours (Tailwind):** Active/Resolved: `bg-green-100 text-green-800 dark:bg-green-900/30 dark:text-green-300`; Upcoming: `bg-blue-100 text-blue-800 dark:bg-blue-900/30 dark:text-blue-300`; Checked Out: `bg-gray-100 text-gray-600 dark:bg-gray-700 dark:text-gray-300`; Cancelled: `bg-red-100 text-red-700 dark:bg-red-900/30 dark:text-red-400`; Submitted: `bg-yellow-100 text-yellow-800 dark:bg-yellow-900/30 dark:text-yellow-300`; In Progress: `bg-orange-100 text-orange-800 dark:bg-orange-900/30 dark:text-orange-300`
- **Loading spinner:** CSS `border-4 border-brand/20 border-t-brand rounded-full w-10 h-10 animate-spin`
- **EmptyState:** Muted icon colour (`text-gray-400 dark:text-gray-500`); title in `text-gray-700 dark:text-gray-200` at `text-base font-semibold`; subtitle in `text-sm text-gray-500 dark:text-gray-400`

## Technical Notes

- **File structure:**
  ```
  js/components/
    toast.component.js
    modal.component.js
    status-badge.component.js
    loading-state.component.js
    empty-state.component.js
    reservation-context-header.component.js
  ```
- **Toast implementation (Alpine.js store approach):**
  ```js
  Alpine.store('toast', {
    items: [],
    show(message, type = 'info') {
      const id = Date.now();
      this.items.push({ id, message, type });
      setTimeout(() => this.dismiss(id), 4000);
    },
    dismiss(id) { this.items = this.items.filter(t => t.id !== id); }
  });
  ```
  Rendered in `index.html` with `x-for="toast in $store.toast.items" :key="toast.id"`
- **Modal implementation:** `Alpine.store('modal', { open: false, title: '', body: '', onConfirm: null })`; triggered via `Alpine.store('modal').open = true`; body content passed as HTML string or via a named template approach
- **StatusBadge:** Implemented as an Alpine.js `x-data` component or a pure HTML template with Tailwind; exposed as a reusable `<template>` or a JS function that returns an HTML string
- **ReservationContextHeader:** Bound to `Alpine.store('reservations').active`; uses `x-effect` or `$watch` to update when active reservation changes
- **All components must be registered** before `Alpine.start()` is called in `js/app.js`

## Test Cases

### Happy Path

1. Call `Alpine.store('toast').show('Booking confirmed', 'success')` → green toast slides in at top-right; auto-dismisses after 4s
2. Call `Alpine.store('toast').show('Network error', 'error')` → red toast appears; manually click `×` → toast dismisses immediately
3. Open modal via `Alpine.store('modal').open = true` on desktop → centred overlay with backdrop; press `Escape` → modal closes
4. Open modal on mobile (375px) → bottom drawer slides up from screen bottom; backdrop click → drawer closes
5. Render `<status-badge status="upcoming">` → blue pill with "Upcoming" label renders
6. Set `Alpine.store('ui').loading = true` → spinner overlay appears; set to `false` → overlay disappears
7. Render `EmptyState` with `icon="wifi_off"`, `title="You're offline"` → correct icon and text displayed

### Edge Cases

- Show 4 toasts in rapid succession — only 3 are shown; oldest is dismissed before 4th renders (or queue scrolls)
- Open a modal, then trigger a toast — toast renders above modal (z-index priority)
- `StatusBadge` receives an unrecognised status string — renders a neutral grey "Unknown" badge without throwing an error
- `ReservationContextHeader` when `$store.reservations.active` is null (no reservations) — component renders nothing or a placeholder; no JS errors
- Modal opened, focus is on an element outside modal — Tab key should not leave the modal until closed
