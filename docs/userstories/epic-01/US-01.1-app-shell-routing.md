## User Story

**As a** guest,
**I want** the app to navigate between pages instantly without full-page reloads,
**So that** the experience feels like a native mobile app rather than a traditional website.

---

## Background / Context

Destinity vButler is a single-page application (SPA) served from a single `index.html` entry point. All navigation is handled client-side using hash-based routing (`window.location.hash`), which avoids server-side routing configuration and works seamlessly with PWA hosting. The router is the foundational piece of the architecture — every other page module depends on it being in place and functioning correctly before any feature work can begin.

---

## Acceptance Criteria

- [ ] `index.html` is the sole HTML file served; all pages are rendered via JavaScript into a `<div id="app">` (or equivalent) mount point
- [ ] A `js/router.js` module exists and exports a `Router` class or init function
- [ ] The router listens to `window.addEventListener('hashchange', ...)` and also runs on initial `DOMContentLoaded`
- [ ] Each registered route maps a hash path (e.g., `#/dashboard`) to a page module import or render function
- [ ] Navigating to an unknown/unrecognised hash redirects to `#/login` when no valid session exists, or `#/dashboard` when a valid session exists
- [ ] All routes except `#/login`, `#/register`, `#/otp`, and `#/forgot-password` require a valid session (checked via `AuthService.checkSession()`); unauthenticated access redirects to `#/login`
- [ ] Browser Back and Forward buttons navigate between previously visited hash routes correctly
- [ ] Page transitions apply a CSS class (e.g., `.page-enter`) that produces a smooth fade or slide animation completing in ≤ 150ms
- [ ] The active route is reflected in the URL bar at all times (hash updates on programmatic navigation)
- [ ] A `Router.navigate(path)` helper function is exposed for programmatic navigation from any module

---

## UI / UX Notes

- Page transition: prefer a fade (`opacity 0 → 1`) or horizontal slide (`translateX`) with `transition-duration: 150ms` and `ease-out` easing
- During transition, the previous page content should not flash — outgoing content fades/slides out before or simultaneously with incoming content
- On first load with no hash, default to `#/login` (unauthenticated) or `#/dashboard` (authenticated)
- The mount point `<div id="app">` should occupy the full viewport; navigation chrome (bottom bar, sidebar) is rendered separately and persisted across route changes to avoid re-rendering
- Loading state: if a page module import is deferred, show a brief spinner in the mount point

## Technical Notes

- **File:** `js/router.js`
- **Pattern:** Hash-based routing using `window.location.hash`; no History API (avoids server config requirements for PWA hosting)
- **Route registry:** Object map `{ '#/login': () => import('./pages/login.js'), '#/dashboard': () => import('./pages/dashboard.js'), ... }`
- **Guard logic:** Before rendering any protected route, call `AuthService.checkSession()` synchronously from `localStorage`; if falsy, call `Router.navigate('#/login')`
- **Rendering approach:** Each page module exports a `render(container)` function that injects HTML into the mount point and initialises its Alpine.js component via `Alpine.initTree(container)` or equivalent; alternatively use `x-show` / `x-if` blocks toggled by Alpine store state
- **Transition implementation:** Toggle a CSS class `.is-transitioning` on the mount point before and after render; CSS handles the animation
- **Dynamic import:** Use `import()` (dynamic) for page modules to enable code splitting if bundling is introduced later; for Phase 1 (no bundler), all scripts may be loaded upfront and the router calls render functions from a registry
- **Alpine.js integration:** After injecting new HTML, call `Alpine.initTree(document.getElementById('app'))` to initialise Alpine directives on newly injected content
- **No bundler in Phase 1:** Scripts loaded via `<script type="module">` in `index.html`; router imported as ES module

## Test Cases

### Happy Path

1. Open `index.html` in browser with no hash and no session — URL becomes `index.html#/login`, login page renders
2. Authenticate successfully — router navigates to `#/dashboard`, dashboard page renders with transition animation
3. Click a nav item (e.g., "Services") — URL updates to `#/services`, services page renders; Back button returns to `#/dashboard`
4. Open `index.html#/dashboard` in a new tab with a valid session in localStorage — dashboard renders directly, no redirect to login
5. Open `index.html#/dashboard` in a new tab with no session — redirects to `#/login`

### Edge Cases

- Navigate to `index.html#/nonexistent` with valid session → redirects to `#/dashboard`
- Navigate to `index.html#/nonexistent` without session → redirects to `#/login`
- Rapid successive hash changes (clicking nav items quickly) — only the final destination renders; no half-rendered states or memory leaks
- Refresh page on `#/services` with valid session — services page renders correctly (not blank)
- Navigate programmatically via `Router.navigate('#/profile')` — same behaviour as clicking a link
- Alpine.js directives on injected page HTML initialise correctly after `Alpine.initTree()` call
