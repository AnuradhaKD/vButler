## User Story

**As a** guest,
**I want** to switch between light and dark display modes,
**So that** I can comfortably use the app in a dark hotel room at night without being blinded by a bright screen.

---

## Background / Context

Hotel guests frequently use their phones in low-light environments — in bed, at dinner, or in a dimly lit room. A dark mode reduces eye strain and battery consumption on OLED displays. The theme preference must persist across sessions so guests don't need to re-apply their preference every time they open the app. On first launch, the app respects the device's operating system preference, creating a seamless out-of-the-box experience.

---

## Acceptance Criteria

- [ ] A theme toggle control is present in the top header (accessible from every authenticated page)
- [ ] The same theme toggle is also accessible from the Settings page
- [ ] Toggling theme switches the `dark` class on the `<html>` element: present = dark mode, absent = light mode
- [ ] The selected theme is saved to `localStorage` under the key `vb:theme` with value `'dark'` or `'light'`
- [ ] On every app load, `ThemeService.init()` is called before first render; it reads `vb:theme` from `localStorage` and applies the correct class to `<html>`
- [ ] If `vb:theme` is not set (first visit), the app checks `window.matchMedia('(prefers-color-scheme: dark)').matches` and applies the appropriate theme
- [ ] All UI surfaces (backgrounds, text, borders, inputs, modals, cards, navigation) have correct dark mode variants via Tailwind `dark:` classes
- [ ] The toggle icon shows a sun (`light_mode`) icon in dark mode and a moon (`dark_mode`) icon in light mode, indicating the action that will be taken
- [ ] Theme change is instant (no loading state, no page reload)
- [ ] `ThemeService` exposes a reactive Alpine.js store property `$store.ui.theme` that other components can bind to

---

## UI / UX Notes

- Toggle control in header: icon-only button (sun/moon Material Symbol); include `aria-label="Switch to light mode"` / `"Switch to dark mode"` for accessibility
- In Settings: a full labelled row with "Dark Mode" label and a toggle switch (`<input type="checkbox">` styled as a pill toggle)
- Avoid a flash of incorrect theme on page load (FOIT/FOUC): the `ThemeService.init()` call must be in a `<script>` tag in `<head>` (inline, not deferred) so the class is applied before the DOM paints
- Dark mode background: `#0f1e23` (`surface.dark`) for page background; cards and surfaces use `dark:bg-brand-dark` (`#002838`) or `dark:bg-gray-800` where appropriate
- Ensure all text maintains sufficient contrast in dark mode: white or `gray-100` on dark backgrounds, never low-contrast grey on grey

## Technical Notes

- **File:** `js/core/theme.service.js`
- **ThemeService implementation:**
  ```js
  const ThemeService = {
    init() {
      const saved = localStorage.getItem('vb:theme');
      const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
      const isDark = saved ? saved === 'dark' : prefersDark;
      document.documentElement.classList.toggle('dark', isDark);
      return isDark ? 'dark' : 'light';
    },
    toggle() {
      const isDark = document.documentElement.classList.toggle('dark');
      localStorage.setItem('vb:theme', isDark ? 'dark' : 'light');
      if (window.Alpine) Alpine.store('ui').theme = isDark ? 'dark' : 'light';
    },
    current() {
      return document.documentElement.classList.contains('dark') ? 'dark' : 'light';
    }
  };
  ```
- **FOUC prevention:** Place inline `<script>` in `<head>` that calls `ThemeService.init()` synchronously (this script must be before Tailwind CDN and before `<body>`)
- **Alpine.js store:** In `js/app.js`, register `Alpine.store('ui', { theme: ThemeService.init(), ... })`; the `init()` return value pre-populates the store at startup
- **OS preference change listener (optional P2):** `window.matchMedia('(prefers-color-scheme: dark)').addEventListener('change', e => { if (!localStorage.getItem('vb:theme')) ThemeService.toggle(); })`
- **Tailwind dark mode:** Configured as `darkMode: 'class'` in `tailwind.config` (US-01.2). All component dark variants use `dark:` prefix classes

## Test Cases

### Happy Path

1. First visit, OS in dark mode — app loads in dark mode; `vb:theme` not yet set; `prefers-color-scheme` respected
2. Click the toggle in the header — theme switches to light mode; `<html>` loses `.dark` class; `vb:theme` = `'light'` in localStorage
3. Refresh page — light mode persists; `vb:theme` = `'light'` read on load
4. Click toggle again — dark mode restored; `vb:theme` = `'dark'`
5. Navigate to Settings — toggle switch reflects the current theme state
6. Toggle from Settings — theme switches globally; header icon updates in sync

### Edge Cases

- First visit, OS in light mode — app loads in light mode; no `vb:theme` key in localStorage
- `vb:theme` localStorage key manually deleted between sessions — falls back to OS preference
- Toggle clicked rapidly 10 times — theme toggles correctly each time; no desync between `<html>` class and Alpine store
- Theme toggled while on the Login page (pre-auth) — toggle still works on auth pages; `vb:theme` persists for post-auth use
- Dark mode active when PWA is installed and launched standalone — dark mode state preserved in standalone context (localStorage persists)
