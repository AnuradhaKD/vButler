## User Story

**As a** guest,
**I want** a clear, persistent navigation system that adapts to my device size,
**So that** I can move between the main sections of the app quickly and always know where I am.

---

## Background / Context

Navigation is the skeleton of Destinity vButler. Guests must be able to reach Home, their Stays, request Services, view their Bill, and access their Profile from anywhere in the app. The bottom tab bar is a well-understood native mobile pattern for thumb reachability; the sidebar is the desktop equivalent. Both must communicate the active section clearly and never appear on authentication screens, which have their own isolated layouts.

---

## Acceptance Criteria

- [ ] On mobile (viewport width < 768px): a fixed bottom tab bar is rendered containing exactly 5 tabs in order — Home, Stays, Services, Bill, Profile
- [ ] On desktop (viewport width ≥ 1024px): a fixed left sidebar is rendered containing all navigation items and the Browns/Destinity vButler logo at the top
- [ ] On tablet (768px–1023px): either the mobile bottom bar or a collapsed sidebar icon rail is shown (define and document the choice)
- [ ] Each navigation item displays the correct Material Symbol Outlined icon (Home → `home`, Stays → `hotel`, Services → `room_service`, Bill → `receipt_long`, Profile → `person`) and a label
- [ ] The active navigation item is visually highlighted: brand colour background pill/indicator + filled icon variant (`font-variation-settings: 'FILL' 1`) + label in brand colour
- [ ] Inactive navigation items display outlined icon variant and muted text colour
- [ ] Tapping/clicking a navigation item navigates to the corresponding route and updates the active state
- [ ] Navigation is NOT rendered on auth pages: `#/login`, `#/register`, `#/otp`, `#/forgot-password`
- [ ] Page content area on mobile has `padding-bottom: 5rem` (pb-20) so content is not obscured by the bottom bar
- [ ] Page content area on desktop has `padding-left` equal to the sidebar width so content is not obscured by the sidebar
- [ ] Navigation items and their routes: Home → `#/dashboard`, Stays → `#/stays`, Services → `#/services`, Bill → `#/bill`, Profile → `#/profile`

## UI / UX Notes

- **Bottom bar (mobile):** Fixed at bottom of viewport; height 56–64px; white/`surface.dark` background; subtle top border (`border-t border-gray-200 dark:border-gray-700`); items evenly distributed with `justify-around`; active item has a pill background in `bg-brand/10` (10% opacity brand) with `text-brand` and filled icon; icons 24px, labels 10–11px
- **Sidebar (desktop):** Fixed left; width 240px; full height; `bg-surface-light dark:bg-surface-dark`; right border `border-r`; logo/wordmark at top (48px tall); navigation items stacked vertically with 8px spacing; active item has a full-width highlighted row with brand colour left border (4px) or rounded pill style; labels visible at all times (no icon-only mode at this width)
- **Icon filled vs outlined:** Outlined (`FILL 0`) for inactive; filled (`FILL 1`) for active — this visual cue reinforces "you are here" in a native-app-idiomatic way
- **Labels:** Always shown on both bottom bar and sidebar (no icon-only mode); keep labels concise (single word)
- **Safe area insets:** On iOS PWA in standalone mode, bottom bar must respect the home indicator safe area; use `padding-bottom: env(safe-area-inset-bottom)` in addition to tab bar height

---

## Technical Notes

- **Component file:** `js/components/nav.component.js` (or inline Alpine component in `index.html`)
- **Alpine.js store integration:** `$store.ui.activeRoute` reflects the current route hash; navigation component binds active state via `:class="$store.ui.activeRoute === '#/dashboard' ? 'active-nav-item' : ''"`
- **Router integration:** On each route change, the router updates `Alpine.store('ui').activeRoute = window.location.hash`
- **Auth page detection:** Navigation component uses `x-show="!$store.auth.isAuthPage"` or the router sets `$store.ui.showNav = false` for auth routes
- **Responsive breakpoints:** Tailwind `md:hidden` on bottom bar (hidden ≥ 768px); `hidden md:block` on sidebar; for tablet, use `hidden lg:block` on sidebar and show bottom bar below 1024px
- **Desktop sidebar width:** Apply `lg:pl-60` (240px) to the main content wrapper so content shifts right of the sidebar
- **iOS safe area:** Add to bottom bar container:
  ```css
  padding-bottom: calc(0.75rem + env(safe-area-inset-bottom));
  ```
- **Accessibility:** Each nav item is a `<button>` or `<a>` with `aria-label` and `aria-current="page"` on the active item

## Test Cases

### Happy Path

1. Open app on a 375px wide mobile screen — bottom tab bar renders with 5 tabs; no sidebar visible
2. Open app on a 1440px wide desktop — sidebar renders with logo and 5 nav items; no bottom bar visible
3. Tap "Services" on mobile bottom bar — `#/services` navigated; Services tab icon becomes filled; label turns brand colour
4. Navigate to login page (`#/login`) — neither bottom bar nor sidebar renders
5. Navigate back to `#/dashboard` after login — navigation reappears
6. On desktop, click "Bill" in sidebar — active state moves to Bill item; previous item deactivates

### Edge Cases

- Screen resized from mobile to desktop during session (browser DevTools resize) — bottom bar hides, sidebar appears; active state preserved
- Active route set to a route with no corresponding nav item (e.g., `#/reservation/123`) — no nav item appears active; no error
- PWA in standalone mode on iOS with home indicator — bottom bar content not obscured by home indicator (`safe-area-inset-bottom` applied)
- Tab bar tapped on already-active tab — no navigation or re-render occurs (or scrolls content to top — document the behaviour)
