## User Story

**As a** guest who has forgotten their password,
**I want** to request a password reset by entering my email address,
**So that** I can regain access to my account without contacting the hotel.

---

## Background / Context

A forgotten password flow is table-stakes for any authenticated application. While Phase 1 does not implement a real email delivery service, the flow must be architecturally complete and visually polished so it can be connected to a backend in Phase 2 with minimal rework. The two-state design (form state → success state) avoids exposing whether an email address is registered, which is an important security consideration for a hospitality app handling guest data.

---

## Acceptance Criteria

- [ ] The forgot password screen is accessible from the Login screen via the "Forgot password?" link
- [ ] **State 1 — Form state:** Centred card layout (max-width 400px) with a lock icon at the top, a heading, a brief instructional subtitle, a single Email Address input (`type="email"`), and a "Send Reset Link" submit button
- [ ] **State 2 — Success state:** Same card layout; lock icon replaced by an envelope icon (`mail` Material Symbol); heading updates to "Check your inbox"; body text: "If an account exists for [user@example.com], a reset link has been sent. Check your spam folder if you don't see it."
- [ ] The email address entered by the user is displayed in the success state message (interpolated into the text)
- [ ] State transition is handled entirely via Alpine.js (e.g., `x-show` / `x-if` toggle on `submitted` boolean) — no page navigation or reload
- [ ] Client-side validation on submit: email must match standard format; if invalid, inline error appears below the field; form does not submit
- [ ] Submit button shows a loading spinner and is disabled while "processing" (simulate 600ms delay with `setTimeout`)
- [ ] After the simulated delay, the component transitions to Success state regardless of whether the email matches a guest record (no account enumeration)
- [ ] **Phase 1 behaviour:** No real email is sent; a console log outputs `"⚠️ DEMO: Password reset requested for [email]. No email sent in Phase 1."`
- [ ] "Back to login" link is present in **both** Form state and Success state; it navigates to `#/login`
- [ ] The "Back to login" link in Success state offers the option to try a different email (navigate to `#/login` and then back to `#/forgot-password`, or directly offer "Try a different email" which resets the component back to Form state)

## UI / UX Notes

- **Card layout:** Centred vertically and horizontally on the page; `max-w-md w-full mx-auto`; `rounded-xl shadow-sm border border-gray-100 dark:border-gray-700`; generous internal padding (p-8 on desktop, p-6 on mobile)
- **Lock icon:** Material Symbol `lock` at 48px, coloured `text-brand`; centered above the heading
- **Heading (Form state):** "Forgot your password?" in Playfair Display; subtitle: "Enter the email address associated with your account and we'll send you a reset link."
- **Envelope icon (Success state):** Material Symbol `mail` at 48px, `text-green-500`; subtle animated appearance (fade or gentle bounce using CSS keyframes)
- **Email in success message:** Display email in `font-semibold` within the paragraph; consider truncating very long emails with `truncate` class if needed
- **Input field:** Full-width, 48px height; same styling as Login/Register fields; focus ring in brand colour
- **Submit button:** Full-width, 48px height, `bg-brand text-white`; loading state with spinner
- **"Back to login" link:** Text-only, `text-brand` with left arrow icon (`arrow_back` Material Symbol, 16px); present below the card content in both states
- **Dark mode:** Card background `dark:bg-surface-dark`; all text colours adapt; icon colours remain consistent

## Technical Notes

- **File:** `js/pages/forgot-password.js`
- **Alpine.js component:**
  ```js
  {
    email: '',
    submitted: false,
    loading: false,
    error: '',
    async submit() {
      this.error = '';
      if (!this.validateEmail(this.email)) {
        this.error = 'Please enter a valid email address.';
        return;
      }
      this.loading = true;
      await new Promise(r => setTimeout(r, 600)); // simulate API delay
      console.warn(`⚠️ DEMO: Password reset requested for ${this.email}. No email sent in Phase 1.`);
      this.loading = false;
      this.submitted = true;
    },
    validateEmail(email) {
      return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
    },
    resetForm() {
      this.submitted = false;
      this.email = '';
      this.error = '';
    }
  }
  ```
- **State management:** Two `<div>` blocks controlled by `x-show="!submitted"` and `x-show="submitted"` respectively; use `x-transition` for the state switch animation
- **"Try a different email" action:** `@click="resetForm()"` — resets component back to form state without page navigation
- **No backend interaction in Phase 1:** The submit handler never makes a network request; the `fetch` call to a password reset endpoint is stubbed with a comment `// TODO Phase 2: POST /api/auth/forgot-password`

## Test Cases

### Happy Path

1. Click "Forgot password?" on Login → arrive at `#/forgot-password`; lock icon and form visible
2. Enter `jane.doe@example.com` and click "Send Reset Link" → button shows spinner for ~600ms → transitions to Success state with envelope icon and message containing `jane.doe@example.com`
3. Console shows `"⚠️ DEMO: Password reset requested for jane.doe@example.com. No email sent in Phase 1."`
4. Click "Back to login" in Success state → navigate to `#/login`
5. Click "Try a different email" in Success state → form resets; email field cleared; back to Form state

### Edge Cases

- Submit with empty email → inline error "Please enter a valid email address"; no spinner
- Submit with invalid format `notanemail@` → inline error; no submission
- Submit with an email not in `data/mock/guests.json` → identical Success state shown (no difference — prevents account enumeration)
- Double-click "Send Reset Link" → second click ignored (button disabled during loading)
- Navigate directly to `#/forgot-password` from browser (not from Login link) → page renders correctly in Form state
- In dark mode: all elements render with correct colour variants; lock icon and envelope icon visible
