## User Story

**As a** returning guest,
**I want** to sign in to Destinity vButler with my email and password,
**So that** I can access my reservation details and hotel services for my stay.

---

## Background / Context

The login screen is the first impression most returning guests will have of Destinity vButler. It must balance Brown Hotels' premium aesthetic with clarity and speed — guests arriving after a long journey want to get in quickly, not navigate a confusing form. The split-panel desktop layout with a brand hero creates a premium feel while maximising form legibility on larger screens. On mobile, the full-screen card approach keeps the design clean and focused.

---

## Acceptance Criteria

- [ ] On desktop (≥ 1024px): split-panel layout — left panel occupies 50% width with a brand gradient background, Browns logo, and tagline; right panel contains the login form
- [ ] On mobile (< 1024px): full-screen layout with Browns logo at top, tagline below, then the login form as a card or inline
- [ ] Form contains: Email Address input (`type="email"`, `autocomplete="email"`) and Password input (`type="password"`, `autocomplete="current-password"`)
- [ ] Password field has a show/hide toggle (eye icon button) that switches `type` between `password` and `text`
- [ ] "Remember me" checkbox is present; when checked, session persists for 30 days; when unchecked, session expires after 24 hours
- [ ] "Forgot password?" link is present below the Password field and navigates to `#/forgot-password`
- [ ] "New guest? Create account" link is present at the bottom and navigates to `#/register`
- [ ] Client-side validation on submit: email must match regex format; password must be non-empty; if either fails, inline error message appears below the respective field (no toast for field errors)
- [ ] On valid credential submission: `AuthService.login(email, password)` is called; matching is performed against `data/mock/guests.json` using SHA-256 hash comparison
- [ ] On successful credential match: navigate to `#/otp`; pass guest identifier to OTP screen state
- [ ] On failed credential match: show error toast "Incorrect email or password"; do not reveal whether email or password was wrong
- [ ] While form is submitting: the submit button shows a spinner and is disabled; all form inputs are disabled
- [ ] Form does NOT submit on Enter key if validation fails

## UI / UX Notes

- **Left hero panel (desktop):** Full-height panel with a CSS gradient from `#003c52` to `#00516e` (top to bottom or subtle diagonal); Browns Hotels logo centred vertically; tagline text in white, Playfair Display, e.g., "Your personal concierge, always at hand."
- **Right form panel (desktop):** White / `surface.light` background; form max-width 400px; vertically centred; "Welcome back" heading in Playfair Display; subtle "Destinity vButler" wordmark above heading
- **Mobile layout:** Full-screen `surface.light` background; Browns logo at top (40–48px); tagline in smaller text; form card with `rounded-xl shadow-sm` and padding; or no card — full-bleed form with generous padding
- **Input fields:** Full-width, height 48px, `rounded-lg border border-gray-300 dark:border-gray-600`; label above field (not floating); focus ring in brand colour `ring-brand`
- **Submit button:** Full-width, height 48px, `bg-brand text-white rounded-lg font-semibold`; hover: `bg-brand-light`; loading state: spinner replaces button label
- **Show/hide password:** Eye icon (`visibility` / `visibility_off` Material Symbol) at right edge of password field; no visual disruption to field layout when toggled

- **Dark mode:** Hero panel remains dark teal; form panel switches to `surface.dark`; inputs use dark mode border and background

## Technical Notes

- **Files:** `js/pages/login.js`, `data/mock/guests.json`
- **AuthService.login(email, password):**
  1. Normalise email to lowercase
  2. Fetch `data/mock/guests.json` (or read from in-memory if already loaded)
  3. Find guest record where `guest.email === email`
  4. Hash the provided password with SHA-256 using `crypto.subtle.digest('SHA-256', ...)` and compare against stored `guest.passwordHash`
  5. Return `{ success: true, guest }` or `{ success: false }`
- **SHA-256 helper:**
  ```js
  async function hashPassword(password) {
    const msgBuffer = new TextEncoder().encode(password);
    const hashBuffer = await crypto.subtle.digest('SHA-256', msgBuffer);
    return Array.from(new Uint8Array(hashBuffer)).map(b => b.toString(16).padStart(2, '0')).join('');
  }
  ```
- **guests.json shape (per record):**
  ```json
  {
    "id": "guest-001",
    "email": "jane.doe@example.com",
    "passwordHash": "<sha256 of 'Password1!'>"
    "fullName": "Jane Doe",
    "phone": "+94771234567",
    "linkedReservationIds": ["res-001", "res-002"]
  }
  ```
- **OTP screen state:** Store `{ guestId, email, rememberMe }` in `sessionStorage` as `vb:pendingAuth` before navigating to `#/otp`; OTP screen reads this and clears it after successful verification
- **Alpine.js component:** `x-data="loginForm()"` with properties `email`, `password`, `showPassword`, `rememberMe`, `submitting`, `errors`
- **Remember me:** Pass `rememberMe` boolean to `AuthService.createSession(guest, rememberMe)` in US-02.5

## Test Cases

### Happy Path

1. Enter valid email (`jane.doe@example.com`) and correct password → button spinner → navigate to `#/otp`; `vb:pendingAuth` written to sessionStorage
2. Check "Remember me" → complete login flow → session `expiresAt` is 30 days from now (verified post-OTP in US-02.5)
3. Uncheck "Remember me" → login → session `expiresAt` is 24 hours from now

### Edge Cases

- Submit with empty email → inline error "Email address is required" below email field; form does not submit
- Submit with invalid email format (`notanemail`) → inline error "Please enter a valid email address"
- Submit with empty password → inline error "Password is required"
- Valid email format but no matching guest record → toast "Incorrect email or password"; no inline field errors
- Correct email but wrong password → same toast; no account enumeration
- Click "Forgot password?" → navigates to `#/forgot-password` without clearing the email field (populate on destination if feasible)
- Click "Create account" → navigates to `#/register`
- Submit button clicked while submitting (double-click) → second submission ignored (button disabled)
- All inputs in dark mode → readable labels, visible borders, no invisible placeholder text
