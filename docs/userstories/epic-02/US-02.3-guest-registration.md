## User Story

**As a** new guest,
**I want** to create a Destinity vButler account with my personal details,
**So that** I can link my reservation and access hotel services for my stay.

---

## Background / Context

First-time guests arrive at registration either from the Login screen's "Create account" link or directly via an invitation from the hotel. The registration form must be frictionless — collecting only the minimum viable information needed to create an account and link to a reservation. The phone number field with country code selector is important for a hotel brand like Browns that serves an international guest base. Password strength feedback encourages guests to create secure credentials from the start.

---

## Acceptance Criteria

- [ ] The registration screen uses the same split-panel layout as Login on desktop (≥ 1024px): brand hero panel left, form right
- [ ] On mobile (< 1024px): full-screen card layout matching the Login mobile layout style
- [ ] Form fields (in order): Full Name (`type="text"`, `autocomplete="name"`), Email Address (`type="email"`, `autocomplete="email"`), Phone Number (with country code prefix selector), Password (`type="password"`, `autocomplete="new-password"`), Confirm Password (`type="password"`, `autocomplete="new-password"`)
- [ ] Phone number field uses `intl-tel-input` library (CDN) for the country code dropdown with flag icons
- [ ] Password field has a show/hide eye toggle
- [ ] Confirm Password field has a show/hide eye toggle independent of the Password field toggle
- [ ] A password strength indicator renders below the Password field, updating in real-time as the user types:
  - Weak: password < 8 chars OR only one character type — red bar (1/3 width), label "Weak"
  - Fair: ≥ 8 chars, 2 character types (e.g., letters + numbers) — amber bar (2/3 width), label "Fair"
  - Strong: ≥ 8 chars, 3+ character types (uppercase, lowercase, number, symbol) — green bar (full width), label "Strong"
- [ ] Terms & Conditions checkbox is present with a link to a terms page or modal; checkbox must be checked before form can be submitted
- [ ] "Already have an account? Sign in" link navigates to `#/login`
- [ ] Client-side validation on submit:
  - [ ] Full Name: required, minimum 2 characters
  - [ ] Email: required, valid format, not already registered in `vb:guests` localStorage
  - [ ] Phone: required (intl-tel-input validates format)
  - [ ] Password: required, minimum 8 characters
  - [ ] Confirm Password: must match Password exactly
  - [ ] Terms checkbox: must be checked
- [ ] Inline field-level errors appear below each invalid field on submit attempt; errors clear when the field is corrected
- [ ] On valid form submission: create a new guest object, hash the password with SHA-256, append to `vb:guests` in localStorage (array), store `vb:pendingAuth` in sessionStorage, navigate to `#/otp`
- [ ] Button shows spinner + disabled state during submission (even though mock — simulate brief 400ms delay for UX)

## Technical Notes

- **File:** `js/pages/register.js`
- **intl-tel-input CDN:**
  - CSS: `https://cdnjs.cloudflare.com/ajax/libs/intl-tel-input/17.0.8/css/intlTelInput.css`
  - JS: `https://cdnjs.cloudflare.com/ajax/libs/intl-tel-input/17.0.8/js/intlTelInput.min.js`
  - Initialise: `intlTelInput(phoneInput, { utilsScript: '...utils.js', preferredCountries: ['lk', 'gb', 'in', 'au', 'us'] })`
- **Password strength logic:**
  ```js
  function getPasswordStrength(password) {
    if (password.length < 8) return 'weak';
    const types = [/[a-z]/, /[A-Z]/, /[0-9]/, /[^a-zA-Z0-9]/].filter(r => r.test(password)).length;
    if (types >= 3) return 'strong';
    if (types === 2) return 'fair';
    return 'weak';
  }
  ```
- **SHA-256 hashing:** Same `hashPassword()` utility as Login (shared in `js/core/auth.service.js`)
- **Guest object shape on registration:**
  ```json
  {
    "id": "guest-<timestamp>",
    "email": "new@example.com",
    "passwordHash": "<sha256>",
    "fullName": "New Guest",
    "phone": "+447712345678",
    "linkedReservationIds": [],
    "createdAt": "<ISO timestamp>"
  }
  ```
- **localStorage write:** Read existing `vb:guests` array (or init `[]`), push new guest, write back to `vb:guests`
- **Duplicate email check:** `vb:guests.find(g => g.email === email.toLowerCase())` — if found, inline error "An account with this email already exists. [Sign in]"
- **Navigation to OTP:** Same as Login — write `vb:pendingAuth` to sessionStorage, navigate `#/otp`
- **Alpine.js component:** `x-data="registerForm()"` with reactive properties for each field, strength, and errors object

## Test Cases

### Happy Path

1. Fill all fields correctly with a new email, strong password, confirm password matches, Terms checked → submit → 400ms delay → navigate to `#/otp`
2. New guest record appears in `vb:guests` localStorage array; `passwordHash` is SHA-256 hash (not plain text)
3. Phone country code selector shows Sri Lanka flag by default; selecting UK shows UK flag + `+44` prefix

### Edge Cases

- Submit with blank Full Name → inline error "Full name is required"
- Full Name with 1 character → inline error "Please enter your full name"
- Invalid email format → inline error "Please enter a valid email address"
- Email already in `vb:guests` → inline error "An account with this email already exists"
- Password shorter than 8 characters → strength indicator shows "Weak"; submit blocked with "Password must be at least 8 characters"
- Confirm Password does not match Password → inline error "Passwords do not match"
- Terms checkbox unchecked → inline error "You must accept the terms to continue"
- Correct all errors after first submit attempt → errors clear as each field is corrected; submit succeeds
- Password show/hide on both password fields toggle independently — revealing one does not reveal the other
- On mobile, intl-tel-input dropdown opens within viewport without overflow
