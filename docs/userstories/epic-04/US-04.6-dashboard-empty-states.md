## User Story

**As a** hotel guest with no active stay,
**I want** the dashboard to gracefully handle situations where I have no current reservation,
**So that** I am not presented with a broken or confusing interface, and I am guided toward booking a future stay with Browns Hotels & Resorts.

---

## Background / Context

Not every guest who opens the Destinity vButler app is currently checked in to a Browns Hotels & Resorts property. A guest may open the app before their first stay, between stays, or after checking out. The dashboard must handle all of these scenarios without displaying empty data placeholders or broken UI components. The Quick Actions grid should remain visible in all states to preserve navigability, while the hero and activity sections adapt contextually.

---

## Acceptance Criteria

- [ ] **State 1 — No active reservation, has upcoming**: the hero section displays the upcoming reservation context (per US-04.2 State B); the rest of the dashboard renders normally
- [ ] **State 2 — No active or upcoming reservation, only past stays**: the hero is replaced with a "No active stays" empty state (see UI/UX Notes); a "Book a stay at Browns" button is displayed; tapping this button opens the chain booking URL from `chain.json → bookingUrl` in the device browser
- [ ] **State 3 — No reservations at all**: identical to State 2 — the "No active stays" empty state and "Book a stay at Browns" button are shown
- [ ] The Quick Actions Grid (US-04.3) is visible and functional in all empty states (States 1, 2, and 3)
- [ ] The Recent Activity Feed (US-04.4) is hidden in States 2 and 3 (no reservation context to filter by)
- [ ] The Upcoming Reservations Teaser Strip (US-04.5) is hidden in States 2 and 3
- [ ] In State 2 and 3, the "No active stays" empty state includes a hotel illustration/icon and a descriptive subheading
- [ ] Subheading text in State 2/3: "You have no active or upcoming stays. Plan your next escape with Browns Hotels & Resorts."
- [ ] The "Book a stay at Browns" button uses the chain's booking URL: `chain.json → bookingUrl`; link opens via `window.open(url, '_blank')`
- [ ] All empty states render correctly in both light and dark mode

## UI / UX Notes

- **State 2 / 3 empty state block** (replaces hero section):
  - Background: `var(--color-surface-card)`, `border-radius: 16px`, `margin: 16px`, `padding: 32px 24px`
  - Hotel illustration: a simple SVG of a hotel building or landmark icon, 80×80px, `color: var(--color-brand-primary)` tint
  - Heading: "No active stays" — Playfair Display, `font-size: 1.25rem`, `font-weight: 700`, centred
  - Subheading: body text, `font-size: 0.875rem`, `color: var(--color-text-secondary)`, centred, max-width 280px, margin auto
  - "Book a stay at Browns" button: brand primary filled button, `border-radius: 8px`, `padding: 12px 24px`, `font-weight: 600`, centred below subheading, margin-top 20px
- **State 1** (no active, has upcoming): no empty state block; hero renders normally with upcoming content from US-04.2; dashboard sections below appear normally
- **Quick Actions grid** in empty states: rendered with the same layout as normal dashboard; if `activeProperty` is null, use `chain.json → defaultProperty` or the most recently visited property for Quick Actions ordering; fallback to the default 8-tile order if no property context is available
- The transition between states should be seamless with no visible layout jumps — use Alpine.js `x-show` with `x-transition` for the empty state block

## Technical Notes

- **State resolution logic** (evaluated on dashboard load and whenever `vb:reservations` changes):
  ```js
  const reservations = StorageService.get('vb:reservations') || [];
  const active = reservations.find(r => r.status === 'active');
  const upcoming = reservations.find(r => r.status === 'upcoming');
  const hasPast = reservations.some(r => r.status === 'checked_out');

  if (active) dashboardState = 'active';
  else if (upcoming) dashboardState = 'upcoming';
  else if (hasPast || reservations.length > 0) dashboardState = 'no_active';
  else dashboardState = 'no_reservations';
  ```
- `dashboardState` is an Alpine.js reactive data property in the root dashboard component (`src/js/pages/dashboard.js`)
- **chain.json structure**: `{ "bookingUrl": "https://www.brownhotels.com/book", ... }`; load via `DataService.getChainConfig()`
- **External link**: `window.open(chain.bookingUrl, '_blank', 'noopener,noreferrer')`
- **Quick Actions without active property**: fall back to `chain.json → defaultPropertyId` to resolve property config, or use hardcoded default tile set
- Use `x-show="dashboardState === 'no_active' || dashboardState === 'no_reservations'"` on the empty state block
- Use `x-show="dashboardState === 'active' || dashboardState === 'upcoming'"` on the Recent Activity Feed and Teaser Strip

## Test Cases

### Happy Path

1. Guest with no active stay but an upcoming reservation (check-in in 4 days): verify hero shows upcoming context, Quick Actions visible, no empty state block shown
2. Guest with only past stays (checked out 10 days ago): verify empty state block renders with hotel illustration, heading, subheading, and "Book a stay at Browns" button; Recent Activity and Teaser Strip hidden
3. Guest with absolutely no reservations: verify same empty state as scenario 2
4. Tap "Book a stay at Browns": verify `chain.json → bookingUrl` opens in a new browser tab

### Edge Cases

- Guest had a reservation but it was cancelled: counts as effectively "no active/upcoming" — empty state shown
- `chain.json` booking URL is missing or null: "Book a stay at Browns" button is hidden or shows a fallback message "Visit our website to book"
- Guest transitions from active stay to checkout while app is open: dashboard should reactively update to no_active state on the next data refresh cycle
- Quick Actions in no_active state: all tiles render; tapping a tile that requires an active stay (e.g., Room Requests) shows the target screen's own "Available during your stay" state
