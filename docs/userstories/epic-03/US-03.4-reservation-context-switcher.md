## User Story

**As a** guest with multiple reservations,
**I want** to switch between my reservations from any page in the app,
**So that** I can manage a current stay while also checking details of an upcoming trip without navigating back to the Stays list each time.

---

## Background / Context

Loyal Browns Hotels guests frequently have simultaneous bookings â€” a current stay at one property and an upcoming reservation at another. The context switcher makes Destinity vButler feel like a true multi-reservation companion rather than a single-booking app. It eliminates friction by allowing switching from any authenticated page, meaning the guest never loses their place in the app just to check which room they'll be in next month.

## Technical Notes

- **File:** `js/components/reservation-context-header.component.js`
- **Alpine.js store (extending from US-03.5):**
  ```js
  Alpine.store('reservations', {
    list: [],          // All guest reservations
    activeId: null,    // Currently selected reservation ID
    get active() {
      return this.list.find(r => r.id === this.activeId) || null;
    },
    setActive(id) {
      this.activeId = id;
      localStorage.setItem('vb:activeReservation', id);
    }
  });
  ```
- **ReservationContextHeader component Alpine data:**
  ```js
  {
    open: false,
    get reservation() { return Alpine.store('reservations').active; },
    get allReservations() { return Alpine.store('reservations').list; },
    get isMulti() { return this.allReservations.length > 1; },
    switchTo(id) {
      Alpine.store('reservations').setActive(id);
      this.open = false;
    }
  }
  ```
- **Reactivity:** Because `active` is a getter on the store, all Alpine components that reference `$store.reservations.active` automatically re-evaluate when `activeId` changes; page content bound to the active reservation (services, bill, etc.) automatically updates
- **Page re-render:** Pages that display active-reservation-specific content must bind to `$store.reservations.active` or `$store.reservations.activeId`; use `x-effect` or Alpine watchers to re-fetch/re-filter data when the active reservation changes
- **Desktop dropdown positioning:** Use `x-anchor` (Alpine Anchor plugin) or manually compute `top` and `left` based on the strip's `getBoundingClientRect()` to avoid viewport overflow
- **LocalStorage sync:** `vb:activeReservation` stores only the reservation `id` (string); on app load, the ReservationService reads this and sets `store.reservations.activeId` accordingly (see US-03.5)

## Test Cases

### Happy Path

1. Log in as guest with 2 reservations â†’ context strip shows the default active reservation (e.g., active or first upcoming); chevron visible on right
2. Tap the strip â†’ bottom drawer opens on mobile with both reservations listed; currently active one highlighted
3. Tap the second reservation in the drawer â†’ drawer closes; strip updates to show second reservation's property name, dates, and status badge
4. `vb:activeReservation` in localStorage updated to the new reservation's ID
5. Services page content (if bound to active reservation) re-renders to reflect the new context without page reload

### Edge Cases

- Guest with only 1 reservation â†’ strip shown without chevron or interaction; no drawer opens on tap
- Guest with 0 reservations â†’ strip shows "No active reservation" placeholder; no switcher; no JS errors
- Guest switches context while viewing Reservation Detail of a different reservation â€” the context strip updates but the detail page continues showing the explicitly-navigated-to reservation (do not auto-redirect the detail view)
- Active reservation is cancelled â†’ strip shows cancelled reservation with "Cancelled" badge; switcher still allows switching to a different active/upcoming reservation
- Switch reservation, navigate to a new page, then back â€” the new active reservation context is preserved; `vb:activeReservation` in localStorage maintains the selection
