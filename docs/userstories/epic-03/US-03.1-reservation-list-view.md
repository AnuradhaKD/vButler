## User Story

**As a** guest,
**I want** to see all of my reservations organised by status,
**So that** I can quickly find my current stay, upcoming bookings, or past visits.

---

## Background / Context

Guests of Browns Hotels may have multiple reservations — a current stay, an upcoming anniversary trip, or past visits they may want to reference for billing. The tabbed list view provides an at-a-glance overview organised by lifecycle stage, reducing cognitive load. Count badges on each tab communicate how many bookings exist in each category before the guest even taps, making navigation predictive and efficient.

---

## Acceptance Criteria

- [ ] The Stays page (`#/stays`) renders a tabbed interface with exactly 4 tabs: Active, Upcoming, Past, Cancelled
- [ ] Tab labels include a count badge when the count for that status is > 0 (e.g., "Active (1)", "Upcoming (2)"); zero-count tabs show just the label with no badge
- [ ] Reservations are filtered into the correct tab based on their `status` field:
  - `active` → Active tab
  - `upcoming` (or `confirmed` with future check-in date) → Upcoming tab
  - `checked_out` or `completed` → Past tab
  - `cancelled` → Cancelled tab
- [ ] The default tab shown on page load is: Active tab if there is ≥ 1 active reservation; otherwise Upcoming tab; otherwise Past tab; otherwise Cancelled tab (priority waterfall)
- [ ] Switching tabs is instant (client-side filter on in-memory data — no re-fetch)
- [ ] When a tab has zero reservations, the tab content area shows an `EmptyState` component with an appropriate icon and message:
  - Active: `hotel` icon, "No active stays. Your next adventure awaits."
  - Upcoming: `event` icon, "No upcoming reservations."
  - Past: `history` icon, "No past stays yet."
  - Cancelled: `cancel` icon, "No cancelled reservations."
- [ ] Reservation cards within each tab are sorted: Active tab — by check-in date ascending; Upcoming tab — by check-in date ascending; Past tab — by check-out date descending (most recent first); Cancelled tab — by check-in date descending
- [ ] Data is sourced from the `vb:reservations` Alpine.js store (hydrated from localStorage; see US-03.5)
- [ ] The page title "My Stays" is shown at the top with the `ReservationContextHeader` strip below it (see US-01.7, US-03.4)

## UI / UX Notes

- **Tab bar:** Horizontally scrollable on mobile if needed; tab items use `border-b-2` underline style for active tab in brand colour; inactive tabs in muted `text-gray-500`; tab font is `font-medium text-sm`
- **Count badge:** Small circular or pill badge in `bg-brand text-white text-xs` positioned as a superscript to the tab label; or inline `(n)` notation in `text-brand font-semibold` — choose one and apply consistently
- **Tab scroll on mobile:** If all 4 tabs don't fit (narrow viewport), allow horizontal scroll on the tab bar with `-webkit-overflow-scrolling: touch`; or use `flex-nowrap overflow-x-auto` with hidden scrollbar
- **Empty state:** Vertically centred in the tab content area; EmptyState component with muted icon; text should feel friendly and reassuring, not clinical
- **Reservation cards:** See US-03.2; render as a vertically stacked list with `gap-4` spacing
- **Loading state:** While reservations are being read from localStorage/store (should be near-instant), show the `LoadingState` spinner briefly; this handles any async initialisation race conditions
- **Page background:** `bg-surface-light dark:bg-surface-dark` for the page; cards have their own white/dark card background

## Technical Notes

- **File:** `js/pages/stays.js`
- **Alpine.js component:**
  ```js
  {
    activeTab: 'active',
    get reservations() { return Alpine.store('reservations').list || []; },
    get active() { return this.reservations.filter(r => r.status === 'active').sort(...); },
    get upcoming() { return this.reservations.filter(r => r.status === 'upcoming').sort(...); },
    get past() { return this.reservations.filter(r => ['checked_out','completed'].includes(r.status)).sort(...); },
    get cancelled() { return this.reservations.filter(r => r.status === 'cancelled').sort(...); },
    init() {
      // Set default tab based on priority waterfall
      if (this.active.length) this.activeTab = 'active';
      else if (this.upcoming.length) this.activeTab = 'upcoming';
      else if (this.past.length) this.activeTab = 'past';
      else this.activeTab = 'cancelled';
    }
  }
  ```
- **Alpine store:** `Alpine.store('reservations')` must expose a `list` array of reservation objects; populated by `ReservationService.loadForGuest()` (US-03.5)
- **Tab switching:** `x-show="activeTab === 'active'"` on each tab content panel; tab button uses `@click="activeTab = 'active'"` and `:class="activeTab === 'active' ? 'border-brand text-brand' : 'border-transparent text-gray-500'"` for active styling
- **Sorting helper:**
  ```js
  const byCheckInAsc = (a, b) => new Date(a.checkIn) - new Date(b.checkIn);
  const byCheckOutDesc = (a, b) => new Date(b.checkOut) - new Date(a.checkOut);
  ```
- **ReservationContextHeader:** Rendered as a component strip at the top of the page content, showing the currently active reservation from `$store.reservations.active`

## Test Cases

### Happy Path

1. Log in as a guest with 1 active + 2 upcoming reservations → Stays page opens on Active tab; Active tab shows "(1)" badge; Upcoming tab shows "(2)" badge; Cancelled tab has no badge
2. Tap "Upcoming" tab → 2 reservation cards appear, sorted by check-in date ascending; Active tab content hides
3. Log in as a guest with no active reservations but 1 upcoming → Stays page opens on Upcoming tab by default
4. Log in as a guest with only past stays → Stays page opens on Past tab; Past tab sorted with most recent first

### Edge Cases

- Guest has 0 reservations in all statuses → Active tab shown by default (first in priority); EmptyState shown; all 4 tabs have no count badge
- Guest has only cancelled reservations → Cancelled tab shown by default; other tabs show EmptyState
- `$store.reservations.list` is undefined or null (before hydration) — component shows `LoadingState`; no JavaScript errors
- A reservation status value is unexpected (e.g., `"pending"`) — it does not appear in any tab (silently filtered out); no error thrown
- Rapidly tapping between tabs — no duplicate renders or flickers; only the selected tab content is shown
