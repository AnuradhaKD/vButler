## User Story

**As a** developer/system,
**I want** reservation data to be loaded once at login and hydrated from localStorage on subsequent app loads,
**So that** the app is fast, works offline, and all feature pages always have access to accurate reservation data without redundant network fetches.

---

## Background / Context

Reservation data is the backbone of virtually every feature in Destinity vButler — navigation context, services, billing, complaints, and pre-arrival all depend on knowing the guest's active reservation. Loading this data reliably and making it universally available via an Alpine.js store is a foundational infrastructure concern. This story defines the data contract (shape and source) and the loading/hydration lifecycle that all consuming feature stories depend on.

---

## Acceptance Criteria

### On Login (Post-OTP)
- [ ] Immediately after `AuthService.createSession()` is called (post-OTP), `ReservationService.loadForGuest(email)` is invoked
- [ ] `loadForGuest(email)` fetches `data/mock/reservations.json` via `DataService.fetch()`
- [ ] The fetched array is filtered to only include reservations where `reservation.guestEmail === email` (case-insensitive match)
- [ ] The filtered reservations array is written to `localStorage` under the key `vb:reservations` (JSON serialised)
- [ ] The active reservation ID is determined using the following priority:
  1. First reservation with `status === 'active'`
  2. First reservation with `status === 'upcoming'` (by soonest check-in date)
  3. First reservation with `status === 'checked_out'` or `status === 'completed'` (most recent)
  4. If none of the above: first item in the list
- [ ] The determined active reservation ID is written to `localStorage` under key `vb:activeReservation`
- [ ] `Alpine.store('reservations').list` is populated with the guest's reservations
- [ ] `Alpine.store('reservations').activeId` is set to the determined active reservation ID
- [ ] Each reservation object in the store includes its complete `folio` (array of charge objects) as a nested property

### On App Load with Valid Session (No Re-Fetch)
- [ ] `ReservationService.hydrateFromStorage()` reads `vb:reservations` from localStorage and parses the JSON
- [ ] The parsed array is loaded into `Alpine.store('reservations').list`
- [ ] `vb:activeReservation` is read from localStorage and loaded into `Alpine.store('reservations').activeId`
- [ ] No network request is made if `vb:reservations` is present in localStorage (offline-safe)
- [ ] If `vb:reservations` is absent (localStorage cleared, first load after session restore) AND a valid session exists: `ReservationService.loadForGuest()` is called as a fallback re-fetch

### Data Service
- [ ] `DataService.fetch(url)` is a utility function that wraps `fetch(url)`; handles the network-first / cache-fallback strategy (coordinating with the service worker from US-01.4)
- [ ] `DataService.fetch()` returns parsed JSON on success
- [ ] `DataService.fetch()` throws an error (or returns null) on failure; callers handle this gracefully

## UI / UX Notes

- This story is primarily a data/service layer story with no direct user-visible UI
- The `LoadingState` spinner (from US-01.7) should be shown briefly during the initial `loadForGuest()` fetch on first login — the overall app loading experience should feel fast (< 500ms for mock JSON)
- If the initial data load fails (network error, no cache), the app should still render but show `EmptyState` on the Stays page with an error message and a "Retry" button that calls `loadForGuest()` again

## Technical Notes

- **Files:**
  - `js/core/reservation.service.js`
  - `js/core/data.service.js`
  - `data/mock/reservations.json`

- **DataService:**
  ```js
  const DataService = {
    async fetch(url) {
      try {
        const response = await fetch(url);
        if (!response.ok) throw new Error(`HTTP ${response.status}`);
        return await response.json();
      } catch (error) {
        // Try service worker cache as fallback
        const cached = await caches.match(url);
        if (cached) return await cached.json();
        throw error;
      }
    }
  };
  ```

- **ReservationService:**
  ```js
  const ReservationService = {
    async loadForGuest(email) {
      const all = await DataService.fetch('/data/mock/reservations.json');
      const mine = all.filter(r => r.guestEmail.toLowerCase() === email.toLowerCase());
      localStorage.setItem('vb:reservations', JSON.stringify(mine));
      const activeId = this._determineActiveId(mine);
      localStorage.setItem('vb:activeReservation', activeId || '');
      this._populateStore(mine, activeId);
      return mine;
    },

    hydrateFromStorage() {
      const raw = localStorage.getItem('vb:reservations');
      const activeId = localStorage.getItem('vb:activeReservation');
      if (!raw) return false;
      const list = JSON.parse(raw);
      this._populateStore(list, activeId);
      return true;
    },

    _determineActiveId(list) {
      const active = list.find(r => r.status === 'active');
      if (active) return active.id;
      const upcoming = list
        .filter(r => r.status === 'upcoming')
        .sort((a, b) => new Date(a.checkIn) - new Date(b.checkIn))[0];
      if (upcoming) return upcoming.id;
      const past = list
        .filter(r => ['checked_out','completed'].includes(r.status))
        .sort((a, b) => new Date(b.checkOut) - new Date(a.checkOut))[0];
      if (past) return past.id;
      return list[0]?.id || null;
    },

    _populateStore(list, activeId) {
      Alpine.store('reservations').list = list;
      Alpine.store('reservations').activeId = activeId;
    }
  };
  ```

- **Alpine store registration (in `js/app.js` before `Alpine.start()`):**
  ```js
  Alpine.store('reservations', {
    list: [],
    activeId: null,
    get active() {
      return this.list.find(r => r.id === this.activeId) || null;
    },
    setActive(id) {
      this.activeId = id;
      localStorage.setItem('vb:activeReservation', id);
    }
  });
  ```

- **Initialisation flow in `js/app.js`:**
  ```
  1. ThemeService.init()
  2. Alpine stores registered (auth, reservations, ui, toast, modal)
  3. Alpine.start()
  4. session = AuthService.checkSession()
  5. if (session) { hydrated = ReservationService.hydrateFromStorage(); if (!hydrated) ReservationService.loadForGuest(session.email); }
  6. Router.init()
  ```

## Test Cases

### Happy Path

1. Login as `jane.doe@example.com` → `loadForGuest()` called → `data/mock/reservations.json` fetched → filtered to Jane's reservations → written to `vb:reservations` → `$store.reservations.list` populated
2. Active reservation ID determined: if Jane has 1 active reservation and 1 upcoming, `activeId` = the active one's ID
3. Close and reopen app (page refresh) with valid session → `hydrateFromStorage()` reads `vb:reservations` from localStorage → store populated without network request
4. Login as a second test guest (`john.smith@example.com`) → only John's reservations appear in the store (Jane's do not)

### Edge Cases

- `data/mock/reservations.json` fetch fails (network down, no cache) → `DataService.fetch()` throws; `loadForGuest()` catches and returns empty array; store populated with `[]`; Stays page shows EmptyState with "Retry" button
- `vb:reservations` in localStorage is malformed JSON → `JSON.parse` wrapped in try/catch; returns `false`; fallback `loadForGuest()` triggered
- Guest has no reservations in `data/mock/reservations.json` → `mine` is `[]`; `vb:reservations` = `"[]"`; `activeId` = null; `$store.reservations.active` = null; all consuming pages handle null gracefully
- Reservation's `folio` array is empty → store contains reservation with `folio: []`; Bill page handles empty folio with EmptyState
- Two guests logged in across tabs (different localStorage contexts) — not a supported scenario in Phase 1; document this limitation
