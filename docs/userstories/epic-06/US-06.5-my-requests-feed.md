## User Story

**As a** hotel guest who has submitted one or more service requests,
**I want** to view all my requests in a live feed with their current status, a visual timeline, and the ability to expand each for full details or cancel if still pending,
**So that** I can track the progress of every request I have made during my stay at Browns Hotels & Resorts without needing to call the front desk for updates.

---

## Background / Context

Real-time request tracking is one of the most operationally differentiating features of the Destinity vButler app for Browns Hotels & Resorts. Guests at luxury boutique hotels have high expectations for service responsiveness and transparency. Providing a live status feed with a visual timeline of each request's journey — from submission through acknowledgement, in-progress, and completion — replaces the frustrating experience of making a phone call to check on a towel request. The cancel capability for pending requests further empowers the guest and reduces unnecessary work for hotel staff.

## Acceptance Criteria

- [ ] The My Requests screen is accessible at route `#/requests`
- [ ] The screen displays all service requests associated with the currently active reservation, filtered by `resvId`
- [ ] Requests are sorted by `submittedAt` descending (most recently submitted first)
- [ ] Two tabs are displayed at the top of the feed: "Active Requests" and "Completed"
- [ ] "Active Requests" tab shows requests with status: `submitted`, `acknowledged`, `in_progress`
- [ ] "Completed" tab shows requests with status: `completed`, `cancelled`
- [ ] Each request card in the feed displays:
  - Service name
  - Reference number (e.g., "REF-20260315-4721") in a monospace label
  - Submitted time as a relative timestamp (Day.js `fromNow()`)
  - Current status badge (colour-coded per US-04.4 status colour rules)
  - Horizontal status timeline with filled/unfilled dots
- [ ] The status timeline shows 4 steps in order: Submitted → Acknowledged → In Progress → Completed; steps up to and including the current status are filled (brand primary); steps beyond current status are unfilled (grey)
- [ ] Tapping a request card expands it (accordion); only one card can be expanded at a time
- [ ] The expanded card area shows:
  - Full request details (all submitted form field values, formatted for readability)
  - Current status note (e.g., "Your request has been received and is being reviewed" for `acknowledged`)
  - A log of timestamps per status change (from `statusHistory[]`), formatted as "DD MMM YYYY HH:mm"
- [ ] A "Cancel Request" button is shown in the expanded area ONLY for requests with status `submitted` or `acknowledged`
- [ ] Tapping "Cancel Request" displays a confirmation prompt: "Are you sure you want to cancel this request?" with "Yes, Cancel" and "Keep Request" buttons
- [ ] Confirming cancellation sets the request status to `cancelled`, `updatedAt` to now, adds to `statusHistory`, and moves the card to the "Completed" (Cancelled) tab
- [ ] When the "Active Requests" tab has no requests, the empty state message is: "No active requests — browse our services to make a request" with a "Browse Services" link to `#/services`
- [ ] When the "Completed" tab has no items, the message is: "No completed or cancelled requests"
- [ ] Relative timestamps update every 60 seconds to stay fresh
- [ ] The feed responds to `vb:request:updated` custom events — when a request status changes via the simulation (US-06.4), the feed updates the affected card without requiring a page reload

## Technical Notes

- **Route**: `#/requests`; navigating to `#/requests/:requestId` should open the same page with the specified card pre-expanded
- **Alpine.js component**: `x-data="myRequestsFeed()"` in `src/js/pages/myRequests.js`
- **Data loading**:
  ```js
  init() {
    this.resvId = StorageService.getActiveReservation()?.id;
    this.loadRequests();
    window.addEventListener('vb:request:updated', () => this.loadRequests());
    this.refreshInterval = setInterval(() => this.tick++, 60 * 1000); // tick increments force Alpine reactivity for fromNow()
  }
  loadRequests() {
    const all = StorageService.get('vb:requests') || [];
    this.requests = all
      .filter(r => r.resvId === this.resvId)
      .sort((a, b) => new Date(b.submittedAt) - new Date(a.submittedAt));
  }
  ```
- **Computed tab data**:
  ```js
  get activeRequests() {
    return this.requests.filter(r => ['submitted','acknowledged','in_progress'].includes(r.status));
  }
  get completedRequests() {
    return this.requests.filter(r => ['completed','cancelled'].includes(r.status));
  }
  ```
- **Timeline step fill calculation**:
  ```js
  const ORDER = ['submitted','acknowledged','in_progress','completed'];
  stepFilled(step, currentStatus) {
    return ORDER.indexOf(step) <= ORDER.indexOf(currentStatus);
  }
  ```
- **Cancellation**:
  ```js
  confirmCancel(requestId) {
    const requests = StorageService.get('vb:requests') || [];
    const req = requests.find(r => r.id === requestId);
    if (!req || !['submitted','acknowledged'].includes(req.status)) return;
    const ts = new Date().toISOString();
    req.status = 'cancelled';
    req.updatedAt = ts;
    req.statusHistory.push({ status: 'cancelled', timestamp: ts });
    StorageService.set('vb:requests', requests);
    this.loadRequests();
    this.showCancelModal = false;
  }
  ```
- **Pre-expanded card**: if route is `#/requests/:requestId`, on init set `openCardId = requestId` and scroll to the card
- **Destroy**: clear `setInterval` and remove event listener on component destroy to prevent memory leaks
- **Status notes map**:
  ```js
  const STATUS_NOTES = {
    submitted: 'Your request has been received.',
    acknowledged: 'Your request has been received and is being reviewed.',
    in_progress: 'Our team is currently handling your request.',
    completed: 'Your request has been completed. We hope you enjoyed the service.',
    cancelled: 'This request was cancelled.'
  };
  ```

## Test Cases

### Happy Path

1. Submit 3 service requests; navigate to `#/requests`
2. Verify all 3 appear in "Active Requests" tab, sorted newest first
3. Verify each card shows service name, reference number, relative time, status badge, and timeline
4. Wait 90 seconds; verify first request's status badge changes to "Acknowledged" and timeline updates — without page reload
5. Tap a request card; verify it expands showing full details, status note, and status log
6. Tap another card; verify first card collapses and new card expands
7. Verify "Cancel Request" button appears in expanded view for a "submitted" request
8. Tap "Cancel Request"; verify confirmation modal appears
9. Tap "Yes, Cancel"; verify request moves to "Completed" tab with "Cancelled" badge
10. Switch to "Completed" tab; verify cancelled request is listed

### Edge Cases

- No active reservation: screen shows "No active requests" empty state; tab bar still renders
- No requests at all for current reservation: "Active Requests" empty state with "Browse Services" link
- "Completed" tab empty: "No completed or cancelled requests" message
- Navigate to `#/requests/REF-ID`: specified card auto-expands on load
- Request in `in_progress` status: "Cancel Request" button is NOT shown in expanded view
- Completed request: no cancel button; status timeline shows all 4 steps filled
- Relative timestamps: "2 hours ago" for a request submitted 2 hours ago; updates every 60 seconds
