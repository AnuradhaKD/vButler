## User Story

**As a** hotel guest who has submitted a service request,
**I want** to receive an immediate, reassuring confirmation with a unique reference number and an estimated delivery time,
**So that** I know my request has been received by Browns Hotels & Resorts and I have a reference I can use to track or enquire about its progress.

---

## Background / Context

Confirmation screens are a critical moment of trust in any service transaction. For a luxury hotel context, the confirmation must feel premium and reassuring — not a generic "submitted" message. The reference number gives guests a concrete identifier they can quote to staff, and the status simulation (via `setTimeout`) demonstrates the app's live-tracking capability even in this prototype implementation, previewing how the experience will feel when connected to a real hotel management backend.

## Acceptance Criteria

- [ ] On form submission from US-06.3, `RequestService.create()` is called with the service object, form data, and active reservation
- [ ] `RequestService.create()` generates a unique reference number in the format `REF-YYYYMMDD-XXXX` where `YYYYMMDD` is today's date and `XXXX` is a zero-padded random 4-digit number
- [ ] The new request object is written to `localStorage` under key `vb:requests` as an array item with the following fields:
  - `id` (UUID or timestamp-based unique ID)
  - `referenceNumber` (e.g., "REF-20260315-4721")
  - `serviceId`
  - `serviceName`
  - `categoryId`
  - `propertyId`
  - `resvId`
  - `status: "submitted"`
  - `details` (the form data object)
  - `submittedAt` (ISO timestamp: `new Date().toISOString()`)
  - `updatedAt` (same as `submittedAt` on creation)
  - `statusHistory: [{ status: "submitted", timestamp: submittedAt }]`
- [ ] The request form content is replaced with the confirmation screen (in-place swap, no page navigation)
- [ ] The confirmation screen displays: a large green animated check circle, the heading "Request Submitted", the reference number prominently displayed, the estimated delivery time, and two buttons: "Track Request" and "Back to Services"
- [ ] "Track Request" navigates to `#/requests/:requestId`
- [ ] "Back to Services" navigates to `#/services`
- [ ] An in-app notification is created immediately: "[Service Name] request received — [Reference Number]"
- [ ] **Status simulation**: after 90 seconds (`setTimeout`), the request status is updated from `submitted` to `acknowledged` in `vb:requests` and a custom event `vb:request:updated` is dispatched
- [ ] **Status simulation**: after 3 minutes (180 seconds), the request status is updated from `acknowledged` to `in_progress` in `vb:requests` and the `vb:request:updated` event is dispatched again
- [ ] The `updatedAt` field and `statusHistory` array are updated on each status change

## Technical Notes

- **RequestService** (`src/js/services/requestService.js`):
  ```js
  create(service, formData, reservation) {
    const now = new Date();
    const dateStr = dayjs(now).format('YYYYMMDD');
    const rand = String(Math.floor(1000 + Math.random() * 9000));
    const referenceNumber = `REF-${dateStr}-${rand}`;
    const id = `req-${Date.now()}-${rand}`;
    const request = {
      id, referenceNumber,
      serviceId: service.id, serviceName: service.name, categoryId: service.categoryId,
      propertyId: reservation.propertyId, resvId: reservation.id,
      status: 'submitted', details: formData,
      submittedAt: now.toISOString(), updatedAt: now.toISOString(),
      statusHistory: [{ status: 'submitted', timestamp: now.toISOString() }]
    };
    const requests = StorageService.get('vb:requests') || [];
    requests.unshift(request);
    StorageService.set('vb:requests', requests);
    this.scheduleStatusSimulation(id);
    NotificationService.create({
      type: 'request_received',
      title: `${service.name} request received — ${referenceNumber}`,
      link: `#/requests/${id}`
    });
    return request;
  }
  ```
- **Status simulation**:
  ```js
  scheduleStatusSimulation(requestId) {
    setTimeout(() => this.updateStatus(requestId, 'acknowledged'), 90 * 1000);
    setTimeout(() => this.updateStatus(requestId, 'in_progress'), 180 * 1000);
  }
  updateStatus(requestId, newStatus) {
    const requests = StorageService.get('vb:requests') || [];
    const req = requests.find(r => r.id === requestId);
    if (!req) return;
    const ts = new Date().toISOString();
    req.status = newStatus;
    req.updatedAt = ts;
    req.statusHistory.push({ status: newStatus, timestamp: ts });
    StorageService.set('vb:requests', requests);
    window.dispatchEvent(new CustomEvent('vb:request:updated', { detail: { requestId, newStatus } }));
  }
  ```
- **Alpine.js**: the confirmation screen is an `x-show="submitted"` block within the same page component as the form; `submitted` is set to `true` after `RequestService.create()` returns
- **Clipboard copy**: `navigator.clipboard.writeText(referenceNumber).then(() => { this.copied = true; setTimeout(() => this.copied = false, 1500) })`

## Test Cases

### Happy Path

1. Submit a valid "Extra Towels" request from US-06.3 form
2. Verify form content replaced by confirmation screen (no page navigation)
3. Verify animated green check circle draws in
4. Verify heading "Request Submitted"
5. Verify reference number in format "REF-YYYYMMDD-XXXX"
6. Verify estimated delivery time shown
7. Verify in-app notification bell badge increments
8. Wait 90 seconds; verify request status updates to "acknowledged" in `vb:requests`
9. Wait 3 minutes; verify status updates to "in_progress"
10. Tap "Track Request"; verify navigation to `#/requests/:id` showing the correct request
11. Tap "Back to Services"; verify navigation to `#/services`

### Edge Cases

- Reference number tap-to-copy on mobile: clipboard receives the reference number; "Copied!" tooltip appears
- Two requests submitted in rapid succession: both generate unique reference numbers; no ID collisions
- App closed/reloaded during the 90s simulation window: `setTimeout` is lost; status remains at "submitted" — this is acceptable for prototype; note in technical docs that a real implementation uses server-side status updates
- `vb:requests` array grows large (50+ items): no performance degradation on storage read/write; consider noting a cleanup strategy for production
- Notification created, but notification bell not visible (e.g., user is on a non-dashboard screen): notification is persisted in `vb:notifications`; bell badge updates when user returns to any screen with the header
