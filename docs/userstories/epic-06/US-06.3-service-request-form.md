## User Story

**As a** hotel guest requesting a hotel service,
**I want** a tailored request form for each service that dynamically presents only the fields relevant to that service,
**So that** I can submit an accurate, complete request with the minimum possible effort and without being confused by irrelevant fields.

---

## Background / Context

Different hotel services have fundamentally different request parameters — a towel request needs a quantity and towel type, a room service order needs a delivery time, and a spa booking needs a preferred date and time. Rather than building separate forms for each of dozens of services, the Destinity vButler app uses a data-driven form field system where each service definition in `property.json` declares its own required fields. This approach allows property managers to configure new services without developer intervention.

## Acceptance Criteria

- [ ] The Service Request Form screen is accessible at route `#/services/:serviceId/request`
- [ ] The form header displays the service name and its category icon
- [ ] A service summary card is displayed at the top of the form showing: description, estimated delivery time, and price (using the same formatting rules as US-06.2)
- [ ] A reservation context bar is displayed showing: "Requesting for: Room [roomNumber], [Property Name]" sourced from the active reservation
- [ ] The form body renders dynamic fields from `service.fields[]` in the order they appear in the array
- [ ] **Supported field types and their rendering**:
  - `preferredTime` → Flatpickr time picker (30-minute increments)
  - `deliveryTime` → Flatpickr time picker (30-minute increments), labelled "Preferred Delivery Time"
  - `quantity` → stepper component: "−" button, numeric display, "+" button; minimum value 1, maximum value 10
  - `specialInstructions` → textarea, max 250 characters, character counter "X / 250"
  - `towelType` → chip multi-select with options: Bath, Pool, Face
  - `date` → Flatpickr date picker, minimum date: today, labelled per `field.label`
- [ ] Each field renders its `field.label` as a label above the input
- [ ] Fields marked `field.required: true` are required before form submission is enabled
- [ ] The "Submit Request" button is displayed at the bottom of the form, full-width, brand primary
- [ ] The "Submit Request" button is disabled when required fields are empty or invalid
- [ ] While submitting, the button shows a loading spinner and is disabled; button text changes to "Submitting..."
- [ ] On successful submission, the form transitions to the confirmation screen (US-06.4)
- [ ] Pressing the device back button or navigation back arrow from this screen returns to the Services Catalog

## Technical Notes

- **Route**: `#/services/:serviceId/request`; router resolves `serviceId`, component loads service from DataService
- **Alpine.js component**: `x-data="serviceRequestForm()"` in `src/js/pages/serviceRequest.js`
- **Service loading**:
  ```js
  async init() {
    const resv = StorageService.getActiveReservation();
    this.reservation = resv;
    this.property = await DataService.getProperty(resv.propertyId);
    this.service = this.property.services.categories
      .flatMap(c => c.services)
      .find(s => s.id === this.serviceId);
    this.fields = this.service.fields || [];
    this.formData = this.initFormData();
  }
  ```
- **Dynamic form data initialisation**:
  ```js
  initFormData() {
    return this.fields.reduce((acc, field) => {
      if (field.type === 'quantity') acc[field.id] = 1;
      else if (field.type === 'towelType') acc[field.id] = [];
      else acc[field.id] = '';
      return acc;
    }, {});
  }
  ```
- **Validation**:
  ```js
  isFormValid() {
    return this.fields
      .filter(f => f.required)
      .every(f => {
        const val = this.formData[f.id];
        if (Array.isArray(val)) return val.length > 0;
        return val !== '' && val !== null;
      });
  }
  ```
- **Submission**: calls `RequestService.create(this.service, this.formData, this.reservation)` which is defined in US-06.4; on success, sets `this.submitted = true` which triggers the confirmation view swap
- **Flatpickr field initialisation**: use `x-init="initFlatpickr($el, field)"` within the field rendering loop; the `initFlatpickr` method selects the right Flatpickr options based on `field.type`
- **Property JSON field definition example**:
  ```json
  {
    "id": "towels",
    "name": "Extra Towels",
    "fields": [
      { "id": "quantity", "label": "Quantity", "type": "quantity", "required": true },
      { "id": "towelType", "label": "Towel Type", "type": "towelType", "required": true },
      { "id": "specialInstructions", "label": "Special Instructions", "type": "specialInstructions", "required": false }
    ]
  }
  ```

## Test Cases

### Happy Path

1. Navigate to `#/services/extra-towels/request` for an active stay in Room 305
2. Verify service name "Extra Towels" in header, category icon shown
3. Verify service summary card shows description, price, estimated time
4. Verify reservation context bar: "Requesting for: Room 305, The Oaks Kandy"
5. Verify quantity stepper starts at 1; tap "+" to 3; tap "−" to 2
6. Verify "−" button disabled at quantity 1
7. Select towel types "Bath" and "Pool"
8. Verify "Submit Request" button is enabled (all required fields filled)
9. Tap "Submit Request"; verify spinner appears; verify transition to confirmation screen

### Edge Cases

- Required field left empty: "Submit Request" button remains disabled
- Quantity stepper at minimum (1): "−" button is visually disabled and non-interactive
- `service.fields[]` is empty or undefined: form renders with no dynamic fields; only the summary card and context bar; Submit button enabled immediately
- Service accessed directly via URL with no active reservation: redirect to `#/services` with "no active stay" state
- Special instructions at 250 chars: counter turns red; no further input accepted
- Service with only a `date` field: form renders with just the date picker; submit enabled once date selected
