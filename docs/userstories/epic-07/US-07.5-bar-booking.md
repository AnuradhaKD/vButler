## User Story

**As a** hotel guest,
**I want** to browse property bars and reserve seating or a table at a bar for a specific date and time,
**So that** I can enjoy cocktails and light bites at the bar without waiting or being turned away during busy periods.

---

## Background / Context

Bars at Browns Hotels & Resorts properties — such as rooftop cocktail bars or poolside bars — are distinct from full-service restaurants in that they typically offer lighter menus (drinks and snacks) and are booked for a session rather than a full dining experience. Surfacing bar listings separately within the Dining Hub, with a simplified booking form, respects this distinction while keeping the guest experience consistent.

---

## Acceptance Criteria

- [ ] Bars with `type: "bar"` in `dining.restaurants[]` appear in the Dining Hub listing with a "Bar" pill instead of a cuisine-type pill
- [ ] Bars are visually distinguishable from restaurants in the Dining Hub list (e.g., distinct pill colour, optional bar icon on card)
- [ ] Tapping a bar card in the Dining Hub navigates to a Bar Detail screen
- [ ] Bar Detail screen uses the same layout as Restaurant Detail (US-07.2): hero image/gradient banner, name in Playfair Display, full hours, short description
- [ ] If `restaurant.menu` is present for a bar, it is rendered with the same accordion layout as US-07.2, but labelled as "Drinks & Snacks" rather than full menu categories
- [ ] If no menu is configured, the menu section is replaced with: "Visit us for cocktails, mocktails, wines, and light bites — curated daily by our bar team"
- [ ] The Bar Detail screen includes a booking form with the following fields: Date (Flatpickr, within stay dates), Time Slot (pills at 30-min intervals from bar hours), Party Size (stepper 1–12), Occasion Note (optional textarea, max 200 chars, placeholder "e.g. Anniversary, birthday celebration")
- [ ] "Confirm Booking" button disabled until date, time, and party size are selected
- [ ] On confirm: a request is written to `vb:requests[]` with `category: "dining"`, `type: "bar_booking"`, and all form field values
- [ ] A success confirmation card displays: bar name, date, time, party size, and unique reference number (`BAR-{timestamp}`)
- [ ] Back navigation returns the guest to the Dining Hub
- [ ] Screen renders correctly in dark mode

---

## UI / UX Notes

- Bar pill in Dining Hub: uses a distinct colour (e.g., deep purple / indigo) to differentiate from cuisine pills; label: "Bar"
- Bar card "Reserve Table" button label changes to "Book a Table" for bars (same navigation target, different label)
- "View Menu" button label changes to "View Bar" for bars
- Bar Detail hero: same 220 px banner with gradient overlay; bar icon (cocktail glass SVG) displayed in top-right corner of banner
- Occasion note field: appears below party size stepper; clearly labelled as optional with "(optional)" in label
- No cancellation policy section required for bars unless `restaurant.cancellationCutoffHours` is present in the JSON (in which case, show as per US-07.3)
- Success card styling identical to US-07.3 success card; reference prefix "BAR-" instead of "TBL-"
- Form inline on Bar Detail page (not a separate screen), consistent with US-07.2's embedded reservation approach

---

## Technical Notes

- Bar Detail reuses the `restaurantDetail()` Alpine.js component; bar-specific differences handled via `restaurant.type === "bar"` conditional logic
- Booking form reuses `tableReservation()` component with `type: "bar_booking"` override and `cancellationCutoffHours` optional
- Reference number: `"BAR-" + Date.now()`
- Request schema mirrors US-07.3 with `type: "bar_booking"` and additional `occasionNote` field:
  ```json
  {
    "id": "BAR-1700000000000",
    "category": "dining",
    "type": "bar_booking",
    "restaurantId": "bar-001",
    "restaurantName": "Skydeck Rooftop Bar",
    "date": "2026-03-18",
    "time": "20:00",
    "partySize": 4,
    "occasionNote": "Anniversary dinner",
    "status": "confirmed",
    "createdAt": "2026-03-15T10:22:00Z"
  }
  ```
- No seating preference field for bars (removed from form)
- No dietary notes field (replaced by occasion note)

## Test Cases

### Happy Path

1. Guest opens Dining Hub; "Skydeck Rooftop Bar" card shows "Bar" pill in indigo colour
2. Guest taps "View Bar" → Bar Detail screen loads with cocktail glass icon on hero
3. Menu section shows "Drinks & Snacks" accordion with 3 items
4. Guest scrolls to booking form; selects 18 March, 20:00, party of 4; adds occasion note "Anniversary"
5. Taps "Confirm Booking" → success card shows: "Skydeck Rooftop Bar · 18 Mar 2026 · 20:00 · 4 guests · Ref: BAR-1710000000000"
6. Guest taps back → returns to Dining Hub

### Edge Cases

- **Bar with no menu**: Placeholder text "Visit us for cocktails…" shown instead of accordion
- **No cancellationCutoffHours on bar**: Policy notice omitted; no errors
- **Occasion note left blank**: Request submitted without `occasionNote` field (or null); no validation error
- **Bar closed at selected time**: Past time slots disabled; if all slots for today are past, date resets to next available day
- **Party size 1**: Allowed; minus button disabled at 1
- **Dark mode**: Bar pill, hero icon, form fields, success card verified
