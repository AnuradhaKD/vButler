## User Story

**As a** hotel guest,
**I want** to reserve a table at a property restaurant directly from the app by selecting a date, time, and party size,
**So that** I can guarantee my preferred dining slot without having to call or visit the front desk.

---

## Background / Context

Table reservations are a high-value touchpoint in luxury hospitality. Allowing guests to self-serve reservations through Destinity vButler reduces load on the concierge team and gives guests the confidence of a confirmed booking with a reference number. The reservation form must respect the guest's stay dates, the restaurant's operating hours, and any cancellation policy configured per-outlet, ensuring operationally viable requests are created.

---

## Acceptance Criteria

- [ ] The reservation screen is reachable from: (a) the Dining Hub "Reserve Table" button, (b) the Restaurant Detail embedded section, and (c) deep link `#/dining/reserve?restaurantId={id}`
- [ ] The restaurant name is displayed prominently at the top of the form as context
- [ ] Date picker (Flatpickr) is constrained to the guest's stay dates (`vb:session.checkIn` to `vb:session.checkOut` inclusive)
- [ ] Time slot selector renders as pill buttons at 30-minute intervals derived from `restaurant.hours.open` to `restaurant.hours.close` (e.g., 07:00, 07:30, 08:00 … 22:00)
- [ ] Time slots for the current day that have already passed are disabled and visually greyed
- [ ] Party size is a stepper control (minus / count / plus) with a minimum of 1 and maximum of 12
- [ ] Seating preference is rendered as radio pill buttons with options: Indoor, Outdoor, No Preference
- [ ] A dietary notes / special requests textarea (max 300 chars) with live character counter is included
- [ ] A cancellation policy notice is displayed above the Confirm button, sourced from `restaurant.cancellationCutoffHours`: "Free cancellation up to [N] hours before your reservation"
- [ ] "Confirm Reservation" button is disabled until a date, time, and party size are selected
- [ ] On confirm: a request object is written to `vb:requests[]` in localStorage with `category: "dining"`, `type: "table_reservation"`, and all form field values
- [ ] A success confirmation card is displayed (not a new screen) with: restaurant name, date, time, party size, seating preference, and a unique reference number (format: `TBL-{timestamp}`)
- [ ] The guest can modify or cancel the reservation from My Requests (US-05.x) up to the cancellation cutoff time
- [ ] If the reservation is within the cancellation cutoff window, the "Cancel" action in My Requests is disabled with tooltip "Cancellation window has passed — please contact reception"
- [ ] Screen renders correctly in dark mode

## UI / UX Notes

- Form layout: single-column, card-based sections with subtle dividers
- Date picker: Flatpickr inline calendar or input-triggered dropdown; brand primary colour for selected date
- Time slot pills: 2-column grid on mobile; selected pill uses brand primary filled state; disabled pills use muted background with strikethrough text
- Party size stepper: large tap targets (44 px), minus button disabled at 1, plus disabled at 12; count displayed in 20 px semi-bold
- Seating preference pills: full-width on mobile, flex row on tablet+; selected state matches time slot pills
- Cancellation policy notice: amber-tinted info card with warning icon, 13 px italic text
- Confirm button: brand primary, full-width, 48 px height, "Confirm Reservation" label
- Success card: brand gradient top border (4 px), green checkmark icon, reference number in monospace 16 px bold, action links "View in My Requests" and "Back to Dining"
- Form errors (missing required fields): inline red text beneath the relevant field, form does not submit

## Technical Notes

- Route: `#/dining/reserve?restaurantId={id}`
- Alpine.js component: `x-data="tableReservation()"` — reads `restaurantId` from URL, loads restaurant from `DataService`
- Time slot generation: utility function `generateTimeSlots(openTime, closeTime, intervalMins)` returning array of `{label, value, disabled}` objects; `disabled` flag set if slot is in the past for today's date
- Stay date constraint for Flatpickr: `minDate: session.checkIn`, `maxDate: session.checkOut`
- Reference number generation: `"TBL-" + Date.now()`
- Request object schema:
  ```json
  {
    "id": "TBL-1700000000000",
    "category": "dining",
    "type": "table_reservation",
    "restaurantId": "restaurant-001",
    "restaurantName": "The Verandah",
    "date": "2026-03-18",
    "time": "19:30",
    "partySize": 2,
    "seatingPreference": "Indoor",
    "dietaryNotes": "One guest is gluten-free",
    "status": "confirmed",
    "cancellationCutoffHours": 2,
    "createdAt": "2026-03-15T10:22:00Z"
  }
  ```
- `StorageService.appendRequest(requestObject)` used for persistence
- Cancellation cutoff enforcement: in My Requests, compare `request.date + request.time` minus `cancellationCutoffHours` against `Date.now()`

## Test Cases

### Happy Path

1. Guest taps "Reserve Table" on The Verandah card in Dining Hub
2. Reservation screen loads with "The Verandah" as context heading
3. Guest opens date picker: only dates within stay (16 Mar – 20 Mar 2026) are selectable
4. Guest selects 18 March; time slots from 07:00 to 22:00 render as pills
5. Guest selects 19:30; party size stepper set to 2; seating preference "Indoor" selected
6. Cancellation policy notice: "Free cancellation up to 2 hours before your reservation"
7. Guest taps "Confirm Reservation" → success card appears with reference TBL-1710000000000
8. Guest taps "View in My Requests" → navigates to My Requests with this reservation visible

### Edge Cases

- **Selecting today's date**: Past time slots (e.g., before current time) are greyed out and unselectable
- **No seating preference selected**: "No Preference" treated as default; Confirm button not blocked
- **Party size at boundary**: Minus button disabled at 1; plus button disabled at 12
- **Cancellation cutoff = 0**: Policy notice shows "Cancellation is not permitted once reserved"
- **restaurant.cancellationCutoffHours absent**: Policy notice omitted gracefully
- **Check-in equals check-out**: Only one date available in picker
- **Dietary notes at 300 char limit**: Character counter shows "300/300"; further input blocked
- **Reservation already exists for same restaurant/date/time**: No duplicate prevention (backend concern); new request appended
- **Guest attempts cancel within cutoff window via My Requests**: Cancel button disabled with tooltip
