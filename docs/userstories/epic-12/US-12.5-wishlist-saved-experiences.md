## User Story

**As a** guest,
**I want** to save experiences I am interested in to a wishlist,
**So that** I can easily find and revisit them later without having to browse the full catalogue again.

---

## Background / Context

Guests browsing the Experiences catalogue may not be ready to book immediately — they may be exploring options early in their stay or comparing alternatives. The wishlist feature enables guests to shortlist experiences for later consideration, reducing re-discovery friction and supporting a more deliberate booking decision. Wishlist state persists across sessions via localStorage, making it available even after the app is closed and reopened.

---

## Acceptance Criteria

- [ ] A heart icon is displayed on every experience card (in the hub grid and on the featured hero card) and on the Experience Detail page
- [ ] Tapping the heart icon when the experience is **not** saved: fills the icon with brand colour, adds `experience.id` to `vb:wishlist.experienceIds[]` in localStorage, and shows a brief toast notification "Saved to Wishlist"
- [ ] Tapping the heart icon when the experience **is** saved: reverts the icon to outline, removes `experience.id` from `vb:wishlist.experienceIds[]`, and shows a brief toast "Removed from Wishlist"
- [ ] The wishlist state persists across page navigation and full session reloads — on each page load, `vb:wishlist` is read from localStorage and heart icons reflect the correct state
- [ ] The "Saved" filter chip on the Experiences Hub (US-12.1) filters the grid to show only experiences whose `id` is in `vb:wishlist.experienceIds[]`
- [ ] When the "Saved" chip is active and the wishlist is empty, an empty state is shown: a heart illustration (SVG or icon) with the text "Save experiences you're interested in" and a "Browse All" link that deactivates the Saved filter
- [ ] The `vb:wishlist` localStorage key stores a single object: `{ experienceIds: [], placeIds: [] }` — both EPIC-12 and EPIC-13 (Local Places) share this key; operations on `experienceIds` must not affect `placeIds`
- [ ] The wishlist toggle is available and functional regardless of reservation status (guests can wishlist even before or after their stay)

## UI / UX Notes

- Heart icon specs: Material Icons `favorite` (filled) and `favorite_border` (outline); 24px; positioned top-right of card image area and in the Detail page header/hero
- Filled state: brand primary colour (e.g. `var(--brand-primary)` or a dedicated `#D4A853` gold for Browns)
- Toggle animation: CSS keyframe scale pulse — `scale(1.3)` at 50%, `scale(1.0)` at 100%; duration 200ms; triggered on every toggle
- Toast notification: appears at top or bottom of screen; auto-dismisses after 2 seconds; dark background, white text, subtle rounded corners; does not overlap important UI elements
- Empty wishlist state: vertically centred within the filtered grid area; SVG heart illustration (outlined, brand colour); 16px muted text; "Browse All" is a text link in brand colour
- On the Detail page, the heart is in the top-right of the hero image at all times (not just on hover); same filled/outline logic applies
- Dark mode: filled heart remains brand colour; outline heart uses `currentColor` (light in dark mode)

## Technical Notes

- **StorageService** (shared utility function or Alpine.js store) for `vb:wishlist` CRUD:
  ```js
  const StorageService = {
    getWishlist() {
      return JSON.parse(localStorage.getItem('vb:wishlist') || '{"experienceIds":[],"placeIds":[]}');
    },
    toggleExperience(id) {
      const w = this.getWishlist();
      const idx = w.experienceIds.indexOf(id);
      if (idx === -1) w.experienceIds.push(id);
      else w.experienceIds.splice(idx, 1);
      localStorage.setItem('vb:wishlist', JSON.stringify(w));
      return w.experienceIds;
    },
    isExperienceSaved(id) {
      return this.getWishlist().experienceIds.includes(id);
    }
  };
  ```
- **Alpine.js** global store (`Alpine.store('wishlist', { ids: [], ... })`) is recommended to ensure all components (hub, card, detail) share reactive wishlist state without re-reading localStorage on every component mount
  - On app init: `Alpine.store('wishlist').ids = StorageService.getWishlist().experienceIds`
  - Toggle method: calls `StorageService.toggleExperience(id)` and updates `Alpine.store('wishlist').ids`
  - Heart icon binding: `x-bind:class="{ filled: $store.wishlist.ids.includes(exp.id) }"`
- Toast implementation: lightweight Alpine.js-driven component; `x-show` with `x-transition`; auto-dismiss via `setTimeout(closeToast, 2000)`
- **Shared key safety**: always read the full `vb:wishlist` object before writing; only modify `experienceIds`; never overwrite `placeIds` unintentionally

## Test Cases

### Happy Path
1. Fresh session: all heart icons are outline; `vb:wishlist` is empty or absent in localStorage
2. Guest taps heart on "Whale Watching Safari" card → icon fills brand colour; toast "Saved to Wishlist" appears and auto-dismisses
3. Guest navigates to hub, taps "Saved" chip → grid shows "Whale Watching Safari" only
4. Guest navigates to Experience Detail for "Whale Watching Safari" → heart icon at top of detail page is filled
5. Guest taps heart on detail page → icon reverts to outline; toast "Removed from Wishlist"
6. Guest returns to hub, taps "Saved" chip → empty state shown
7. Guest closes and reopens app → wishlist state matches last known state (persisted in localStorage)

### Edge Cases
- **`vb:wishlist` key missing in localStorage**: `getWishlist()` returns default `{ experienceIds: [], placeIds: [] }`; no error; save operation creates the key
- **Multiple tabs/windows**: localStorage changes in one tab update correctly when other tabs re-read on focus (note: reactive sync across tabs is not required in Phase 1)
- **Wishlist with many items (20+)**: Filtering and rendering complete without noticeable lag
- **Toggle while "Saved" filter is active**: Guest unsaves an experience while in Saved view → card disappears immediately (Alpine.js reactive); `wishlistIds` count updates
- **`placeIds` preserved**: Toggle an experience → read `vb:wishlist` → confirm `placeIds` array is unchanged
- **Wishlist toast overlap**: Rapid successive toggles → each toast queues/replaces correctly; no visual stacking
- **Experience detail page loaded directly via URL**: Wishlist state still loaded from localStorage on component init
