## User Story

**As the** Destinity vButler system,
**I want** to trigger contextual in-app notifications at the correct moments throughout the guest journey,
**So that** guests at Browns Hotels & Resorts are kept informed about every important event without needing to check each screen manually.

---

## Background / Context

The guest journey at Browns Hotels & Resorts spans pre-arrival (days before check-in), active stay (from check-in to check-out), and post-stay (feedback). Each phase has distinct events that carry high informational value for the guest — a request status change, a billing charge, or a wake-up call confirmation. This story defines exactly when each notification is created, what the opt-out rules are, and how pre-arrival time-based triggers are evaluated in a client-side app without server-side cron jobs.

## Acceptance Criteria

### Mandatory Notifications (no opt-out)

- [ ] **Service request submitted:** `NotificationService.create()` is called immediately when a guest submits any service request (room service, housekeeping, transport, dining, wake-up call, etc.). Type: `serviceRequest`. `linkTo`: `#/requests`. Cannot be opted out via notification preferences.
- [ ] **Service request status change:** A notification is created whenever the status of a service request changes (e.g., Pending → In Progress → Completed). Type: `serviceRequest`. `linkTo`: `#/requests/{id}`.
- [ ] **Complaint submitted:** A notification is created immediately on complaint submission. Type: `complaint`. `linkTo`: `#/complaints`. Cannot be opted out.
- [ ] **Complaint status change:** A notification is created whenever a complaint status updates. Type: `complaint`. `linkTo`: `#/complaints/{id}`.
- [ ] **Payment confirmed:** A notification is created when a payment is processed successfully. Type: `billing`. `linkTo`: `#/billing`. Cannot be opted out.
- [ ] **Wake-up call confirmed:** A notification is created when a wake-up call is scheduled. Type: `wakeUp`. `linkTo`: `#/services/wake-up`. Cannot be opted out.

### Opt-Out Notifications (checked against `vb:profile.notificationPrefs`)

- [ ] **Billing charge added:** A notification is created when a charge is added to the bill. Type: `billing`. `linkTo`: `#/billing`. Only created if `notificationPrefs.billingAlerts === true`.
- [ ] **Pre-arrival checklist incomplete (48h before check-in):** If the guest's check-in date is within 48 hours AND the pre-arrival checklist is not fully complete, a notification is created. Type: `checkIn`. Only created if `notificationPrefs.checkInReminders === true`.
- [ ] **Check-in reminder (24h before):** If the guest's check-in date is within 24 hours, a notification is created. Type: `checkIn`. Only created if `notificationPrefs.checkInReminders === true`.
- [ ] **Post-stay feedback prompt (on check-out):** A notification is created when the guest checks out, prompting them to leave a review. Type: `feedback`. `linkTo`: `#/feedback`. Only created if `notificationPrefs.promotions === false` does not block it — specifically checked against `notificationPrefs.serviceUpdates` (feedback is considered a service update).

### Time-Based Trigger Mechanism (No Cron)

- [ ] The pre-arrival (48h) and check-in (24h) reminders are evaluated **on every app open** (when the authenticated dashboard loads).
- [ ] A deduplication check prevents the same reminder from being created more than once: before creating, check if a notification of the same type and `resvId` already exists in `vb:notifications[]` created within the last 25 hours.
- [ ] If the deduplication check passes and conditions are met, the notification is created once.

### Opt-Out Enforcement

- [ ] Before creating any opt-out notification, read `StorageService.get('vb:profile').notificationPrefs` and check the relevant boolean.
- [ ] If the relevant `notificationPrefs` key is `false`, `NotificationService.create()` is NOT called.
- [ ] Mandatory notifications (listed above) bypass the `notificationPrefs` check entirely.

## Technical Notes

- Each trigger is implemented as a call to `NotificationService.create()` at the appropriate location in the relevant service/module:

  | Trigger | Location in Codebase |
  |---------|----------------------|
  | Service request submitted | `RequestService.submit()` |
  | Service request status change | `RequestService.updateStatus()` |
  | Complaint submitted | `ComplaintService.submit()` |
  | Complaint status change | `ComplaintService.updateStatus()` |
  | Billing charge added | `billing.js addCharge()` |
  | Payment confirmed | `billing.js processPayment()` |
  | Pre-arrival 48h | `dashboard.js onMount()` |
  | Check-in 24h reminder | `dashboard.js onMount()` |
  | Post-stay feedback | `checkout.js onCheckout()` |
  | Wake-up call confirmed | `WakeUpService.schedule()` |

- Opt-out helper:
  ```js
  function canNotify(prefKey) {
    const prefs = StorageService.get('vb:profile')?.notificationPrefs;
    if (!prefs) return true; // default: all on
    return prefs[prefKey] !== false;
  }
  ```
- Pre-arrival deduplication:
  ```js
  function hasRecentNotification(type, resvId, withinHours = 25) {
    const list = StorageService.get('vb:notifications') || [];
    const cutoff = new Date(Date.now() - withinHours * 60 * 60 * 1000).toISOString();
    return list.some(n => n.type === type && n.resvId === resvId && n.createdAt > cutoff);
  }
  ```
- Dashboard `onMount` check:
  ```js
  const reservations = StorageService.get('vb:reservations') || [];
  const now = new Date();
  reservations.forEach(resv => {
    const checkIn = new Date(resv.checkInDate);
    const hoursUntil = (checkIn - now) / 3600000;
    if (hoursUntil > 0 && hoursUntil <= 48 && canNotify('checkInReminders') && !hasRecentNotification('checkIn', resv.id, 25)) {
      NotificationService.create({ type: 'checkIn', title: 'Complete Your Pre-Arrival', body: `...`, resvId: resv.id, linkTo: '#/pre-arrival' });
    }
    if (hoursUntil > 0 && hoursUntil <= 24 && canNotify('checkInReminders') && !hasRecentNotification('checkIn', resv.id, 23)) {
      NotificationService.create({ type: 'checkIn', title: 'Check-In Tomorrow!', body: `...`, resvId: resv.id, linkTo: '#/pre-arrival' });
    }
  });
  ```

## Test Cases

### Happy Path

1. Submit a room service request for extra towels → verify "Request Received" notification appears in `vb:notifications[]` and badge count increments.
2. Update the request status to "In Progress" → verify "Request Update" notification created.
3. Submit a complaint → verify "Complaint Logged" notification created.
4. Set `notificationPrefs.billingAlerts = true` → add a billing charge of LKR 1,500 → verify "Charge Added" notification created.
5. Set check-in date to 20 hours from now → open dashboard → verify "Check-In Tomorrow!" notification created once.
6. Close and reopen app (dashboard mount fires again) → verify deduplication prevents a second "Check-In Tomorrow!" notification.
7. Check out → verify "How Was Your Stay?" feedback notification created.
8. Schedule a wake-up call for 7:00 AM tomorrow → verify "Wake-Up Call Set" notification created.

### Edge Cases

- **`notificationPrefs.billingAlerts = false`:** Add a charge → verify NO "Charge Added" notification created.
- **`notificationPrefs.checkInReminders = false`:** Check-in in 10 hours → open dashboard → verify NO check-in reminder notification.
- **Check-in in 50 hours:** Dashboard mount → verify no pre-arrival or check-in reminder triggered.
- **Check-in in 30 hours (within 48h but not 24h):** Only the 48h pre-arrival notification created; not the 24h reminder.
- **Multiple reservations:** Each reservation evaluated independently; notifications created per `resvId`.
- **`notificationPrefs` missing entirely:** Defaults to all enabled; all opt-out notifications are created.
- **Wake-up call — mandatory:** Even if `notificationPrefs.serviceUpdates = false`, the wake-up call notification is created.
- **Payment confirmed — mandatory:** Even if all prefs are off, "Payment Confirmed" is always created.
