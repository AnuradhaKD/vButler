## User Story

**As a** hotel guest,
**I want** to receive browser push notifications when important events occur,
**So that** I am alerted even when the Destinity vButler app tab is not actively in focus, without needing to install a native mobile app.

---

## Background / Context

Guests at Browns Hotels & Resorts often browse other tabs while waiting for room service or housekeeping. Browser push notifications allow Destinity vButler to surface time-sensitive updates — like a request moving to "Completed" or a billing charge being added — without requiring the guest to keep the app tab active. Phase 1 scope is limited to notifications visible while the browser is open (no VAPID server push), making this a pure client-side enhancement on top of the existing in-app notification system.

---

## Acceptance Criteria

- [ ] After the dashboard screen loads (not immediately on login), the app requests browser notification permission using `Notification.requestPermission()`.
- [ ] The permission prompt is triggered at most once — only when the current permission state is `"default"` (not yet decided). If permission is `"granted"` or `"denied"`, the prompt is not shown.
- [ ] The permission request is NOT shown on the login, register, OTP, or forgot-password screens; only on authenticated screens starting with the dashboard.
- [ ] If the guest grants permission (`"granted"`), subsequent calls to `NotificationService.create()` also trigger `self.registration.showNotification(title, options)` via the service worker.
- [ ] If the guest denies permission (`"denied"`), in-app notifications continue to work normally and the permission prompt is NEVER shown again during the current session or future sessions.
- [ ] The browser push notification mirrors the in-app notification exactly: same `title` and `body` text.
- [ ] The push notification includes the Destinity vButler / Browns Hotels & Resorts app icon.
- [ ] Clicking a browser push notification focuses the Destinity vButler app tab and navigates to the `linkTo` route from the notification data.
- [ ] Phase 1 scope: push notifications function only while the browser is open (tab may be backgrounded). No VAPID server push, no background delivery when browser is closed.
- [ ] The service worker (`/sw.js`) is registered on app boot and handles `push` and `notificationclick` events.
- [ ] In-app notifications (bell badge, notification history) are unaffected by whether push permission is granted or denied.

---

## UI / UX Notes

- Permission prompt timing: triggered 3–5 seconds after the dashboard `x-init` fires (use a `setTimeout`) to avoid the prompt appearing the instant the page loads, which is a known UX anti-pattern that reduces grant rates.
- No custom pre-permission dialog is required in Phase 1 (the native browser permission prompt is used directly).
- Future enhancement (Phase 2): show a branded in-app pre-permission explanation card ("Stay informed about your requests — allow notifications") before triggering the OS prompt.
- Browser push notification appearance (Chrome desktop):
  - Icon: 192×192px Destinity vButler / Browns Hotels & Resorts logo PNG.
  - Title: same as in-app notification title.
  - Body: same as in-app notification body.
  - Badge icon: 72×72px monochrome icon (for Android notification tray).
- The notification should not vibrate or make sound by default (silent push) to avoid startling guests.
- On mobile browsers (Android Chrome): push notifications appear in the notification tray if the browser is in the background.

## Technical Notes

- Service worker registration (in `app.js` or `main.js`):
  ```js
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.register('/sw.js').then(reg => {
      window._swRegistration = reg;
    });
  }
  ```
- Permission request (in `dashboard.js`, after `x-init` settles):
  ```js
  setTimeout(() => {
    if ('Notification' in window && Notification.permission === 'default') {
      Notification.requestPermission().then(permission => {
        StorageService.set('vb:pushPermission', permission);
      });
    }
  }, 4000);
  ```
- `NotificationService.create()` enhanced to optionally fire a browser notification:
  ```js
  if (Notification.permission === 'granted' && window._swRegistration) {
    window._swRegistration.showNotification(title, {
      body,
      icon: '/assets/icons/icon-192.png',
      badge: '/assets/icons/badge-72.png',
      data: { linkTo },
      silent: true
    });
  }
  ```
- Service worker `notificationclick` handler (`/sw.js`):
  ```js
  self.addEventListener('notificationclick', event => {
    event.notification.close();
    const linkTo = event.notification.data?.linkTo || '/';
    event.waitUntil(
      clients.matchAll({ type: 'window', includeUncontrolled: true }).then(clientList => {
        for (const client of clientList) {
          if (client.url.includes(self.location.origin) && 'focus' in client) {
            client.focus();
            client.postMessage({ type: 'NAVIGATE', linkTo });
            return;
          }
        }
        if (clients.openWindow) return clients.openWindow('/' + linkTo);
      })
    );
  });
  ```
- App must listen for `navigator.serviceWorker.addEventListener('message', ...)` to handle `NAVIGATE` messages from the service worker and update the router hash.
- `vb:pushPermission` in localStorage caches the permission result; checked on boot to avoid re-querying `Notification.permission` on every mount.
- Browsers without Notification API (rare, e.g., iOS Safari < 16.4): feature-detect with `'Notification' in window`; skip silently.

## Test Cases

### Happy Path

1. Open Destinity vButler in Chrome desktop with no prior notification permission.
2. Log in and navigate to the dashboard.
3. Wait 4 seconds — verify the browser's native notification permission prompt appears.
4. Click "Allow".
5. Trigger a service request submission.
6. Verify both: (a) in-app notification bell badge increments, AND (b) a browser push notification appears in the top-right of the screen.
7. Click the browser push notification — verify the Destinity vButler tab gains focus and navigates to `#/requests`.
8. Refresh the browser — verify the permission prompt does NOT appear again (state is `"granted"`).

### Edge Cases

- **Guest clicks "Block" on permission prompt:** In-app notifications still work; badge updates; history screen populates. No further permission prompts in this session or future sessions.
- **Guest closes the permission prompt without choosing:** `Notification.permission` remains `"default"`; prompt may appear again on next session (browser-dependent).
- **Notification while Destinity vButler tab is active:** Browser push notification still appears; in-app badge also increments (no duplication in the notification history — push is a display-only layer).
- **Notification while another tab is active (browser backgrounded tab):** Browser push notification appears; on click, Destinity vButler tab is focused.
- **iOS Safari (pre-16.4):** `'Notification' in window` is `false`; no push, no error; in-app works normally.
- **Firefox private browsing:** Service worker registration may fail; feature-detect; fall back gracefully to in-app only.
- **Service worker registration fails:** `window._swRegistration` is `undefined`; `NotificationService.create()` guard `if (window._swRegistration)` prevents crash; in-app notification still created.
- **Multiple browser push notifications queued:** Each fires independently; all appear in browser notification area; clicking any one navigates to its `linkTo`.
- **Dark mode on notification banner:** Browser-rendered; Destinity vButler has no control over OS notification appearance.
