## User Story

**As a** hotel guest,
**I want** to see a notification bell icon with an unread count badge in the app header,
**So that** I am immediately aware of new updates from Browns Hotels & Resorts without having to navigate to a separate screen.

---

## Background / Context

During a stay at a Browns Hotels & Resorts property, guests interact with multiple services — room service requests, housekeeping, spa bookings, and billing. Real-time awareness of status updates reduces anxiety and the need to call reception. The bell badge provides a persistent, low-friction signal that draws attention to new information while remaining unobtrusive when there is nothing new.

---

## Acceptance Criteria

- [ ] A bell icon is visible in the app header on every authenticated page (dashboard, services, dining, profile, settings, etc.).
- [ ] When there are one or more unread notifications, a red circular badge is displayed overlapping the top-right corner of the bell icon, showing the unread count as a number.
- [ ] When the unread count is zero, the badge is hidden (not just zero — fully hidden with `x-show` or `v-if` equivalent).
- [ ] The badge displays the exact unread count for counts 1–99; for counts ≥ 100, it displays "99+".
- [ ] The badge count updates reactively whenever a new notification is added to `vb:notifications[]` via `NotificationService.create()` — no page reload required.
- [ ] The badge count updates reactively when notifications are marked as read — the count decrements accordingly.
- [ ] Tapping the bell icon navigates the user to `#/notifications`.
- [ ] Once on the notifications screen and all notifications are marked as read, returning to any other screen shows a badge count of zero (badge hidden).
- [ ] The bell icon and badge are accessible: the bell button has `aria-label="Notifications"` and `aria-describedby` referencing the badge count for screen readers.
- [ ] The badge is visible in both light and dark mode with sufficient contrast.
- [ ] The bell icon renders correctly in RTL layout (position consistent relative to header regardless of direction).

## UI / UX Notes

- Bell icon: 24px outline-style bell icon (Heroicons or similar); colour: header icon colour (adapts to theme).
- Badge: 18px × 18px minimum circle (expands to pill for 2-digit numbers); background: #EF4444 (red-500); text: white, 11px, font-weight 700.
- Badge position: absolute, top: -4px, right: -4px relative to the bell icon container.
- Tap target for bell button: minimum 44×44px (padding around 24px icon).
- Animation on new notification arrival: badge performs a brief scale pop (scale 1 → 1.3 → 1, 300ms ease) to draw attention.
- Header layout: bell icon grouped with other header actions (e.g., search, profile avatar); consistent 8px spacing between icons.
- RTL: bell icon position within the header group reverses naturally if flex-direction is reversed; the badge always overlaps top-right of the bell icon regardless of layout direction.
- Badge for 99+: pill shape (border-radius: 9999px), text "99+", min-width 24px.

## Technical Notes

- Alpine.js global store: `Alpine.store('notifications')` with a `list` array property and a computed `unreadCount` getter:
  ```js
  Alpine.store('notifications', {
    list: StorageService.get('vb:notifications') || [],
    get unreadCount() {
      return this.list.filter(n => !n.isRead).length;
    }
  });
  ```
- Bell badge HTML pattern:
  ```html
  <button aria-label="Notifications" @click="$router.push('#/notifications')">
    <span class="relative inline-block">
      <!-- Bell icon SVG -->
      <span
        x-show="$store.notifications.unreadCount > 0"
        x-text="$store.notifications.unreadCount > 99 ? '99+' : $store.notifications.unreadCount"
        class="badge-red absolute -top-1 -right-1"
        :aria-label="`${$store.notifications.unreadCount} unread notifications`"
      ></span>
    </span>
  </button>
  ```
- The store must be updated by `NotificationService.create()` and `NotificationService.markRead()` by mutating `Alpine.store('notifications').list` (not replacing the reference, to preserve reactivity).
- `vb:notifications[]` is the localStorage source of truth; the Alpine store is initialised from it on app boot and kept in sync on every mutation.
- `unreadCount` is a getter (not a stored value) so it always derives from the current `list` — no manual count management.
- Header component must be a shared partial included on all authenticated pages; the bell badge should not be duplicated.

## Test Cases

### Happy Path

1. Log in — verify bell icon is present in the header with no badge (zero unread notifications).
2. Trigger a service request submission — verify badge appears with count "1".
3. Navigate between screens (dining, profile, services) — verify badge persists with count "1" on all pages.
4. Trigger two more notifications — verify badge shows "3".
5. Tap the bell — navigate to notifications screen — mark all as read — navigate back to dashboard.
6. Verify badge is hidden (unread count = 0).
7. Refresh the browser — verify badge count is correctly restored from `vb:notifications[]`.

### Edge Cases

- **100 unread notifications:** Badge shows "99+".
- **99 unread notifications:** Badge shows "99" (not "99+").
- **1 notification marked read reduces count from 3 to 2:** Badge updates reactively to "2" without page reload.
- **Dark mode:** Red badge visible against dark header background; bell icon adapts to dark icon colour.
- **RTL (Arabic):** Bell icon position in header correct; badge overlaps top-right of the icon.
- **No notifications array in localStorage:** Badge hidden; no JavaScript error; store initialised with empty array.
- **New notification arrives while user is on notifications screen:** Badge does not flash (notifications screen auto-marks as read on mount — see US-17.2).
