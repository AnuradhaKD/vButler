## User Story

**As a** guest,
**I want** to filter local places by category,
**So that** I can quickly narrow down places that match what I am looking for — whether that is food, nature, culture, or nightlife.

---

## Background / Context

Guests have varied interests and limited time. The category filter empowers guests to self-segment the local places catalogue without scrolling through irrelevant results. For a luxury boutique property like Browns Hotels & Resorts, the curation of categories reflects the breadth of the local experience — from fine dining to adventure activities — and the filter must be intuitive, persistent, and visually consistent.

---

## Acceptance Criteria

- [ ] A horizontal scrollable row of **category filter chips** is displayed below the tab bar on the Local Exploration Hub, in the following order: All | Food | Nature | Culture | Shopping | Adventure | Nightlife
- [ ] "All" chip is selected by default on page load; it is visually distinguished from inactive chips (brand background, white text)
- [ ] Tapping a category chip selects it (brand active style) and filters the visible place cards to only those where `place.category` matches the selected chip label (case-insensitive comparison)
- [ ] Tapping the "All" chip removes the category filter and shows all places (within the current tab's scope)
- [ ] Only one category chip can be active at a time; selecting a new chip deselects the previous one
- [ ] The **category filter state persists while navigating between the Nearby, Popular, and Saved tabs** — switching tabs does not reset the selected category chip
- [ ] If no places match the selected category within the current tab, an empty state is shown with text "No [Category] places found" and a "Clear Filter" text link that resets the category to "All"
- [ ] Category chips that correspond to categories **not present** in the current property's `localPlaces[]` data are not rendered (prevents dead-end filters with zero results on every tab)

## UI / UX Notes

- Chip row container: `overflow-x: auto; white-space: nowrap; -webkit-overflow-scrolling: touch`; no visible scrollbar on mobile
- Individual chip: `display: inline-flex; align-items: center; padding: 6px 14px; border-radius: 20px; font-size: 13px; font-weight: 500`
- Active chip: `background-color: var(--brand-primary); color: #fff; border: 1px solid var(--brand-primary)`
- Inactive chip: `background-color: transparent; color: var(--text-secondary); border: 1px solid var(--border-color)`
- Transition: `background-color 0.15s ease, color 0.15s ease` for smooth activation
- Chip gap: 8px (`gap: 8px` on flex container or `margin-right: 8px` on each chip)
- "All" chip is always rendered regardless of what categories exist in data
- Chip label text matches exactly the category names for consistency: "Food", "Nature", "Culture", "Shopping", "Adventure", "Nightlife"
- Empty filter state: same as hub empty state — icon + message + "Clear Filter" text link in brand colour
- Dark mode: inactive chip border and text use dark-mode variables; active chip brand colour unchanged

## Technical Notes

- Category filter state (`selectedCategory`) lives in the parent `localHub` Alpine.js component (`x-data="localHub()"`) — shared with tab switching logic, ensuring persistence across tabs
- Chip rendering: derive rendered chip list by combining the ordered master list with categories present in `localPlaces`:
  ```js
  const masterCategories = ['Food', 'Nature', 'Culture', 'Shopping', 'Adventure', 'Nightlife'];
  const presentCategories = [...new Set(this.places.map(p => p.category))];
  this.availableCategories = masterCategories.filter(c => presentCategories.includes(c));
  ```
- Filter application (same getter as US-13.1):
  ```js
  if (this.selectedCategory !== 'all') {
    list = list.filter(p => p.category.toLowerCase() === this.selectedCategory.toLowerCase());
  }
  ```
- "Clear Filter" link: `@click="selectedCategory = 'all'"` — no page navigation required
- Chips are rendered via `x-for` over `['All', ...availableCategories]`; `x-bind:class` applies active/inactive styles based on `selectedCategory === chip.toLowerCase()` (with 'all' for the All chip)
- No server call — purely client-side filter on the already-loaded `places` array

## Test Cases

### Happy Path
1. Hub loads; "All" chip is active; all 8 place cards visible on "Nearby" tab
2. Guest taps "Food" chip → "Food" chip activates; only Food-category cards shown (e.g. 3 cards)
3. Guest taps "Nearby" → "Popular" tab switch → "Food" chip remains active; Popular tab shows only Food places
4. Guest taps "All" chip → all places in Popular tab shown
5. Guest taps "Culture" chip → Culture places shown; 0 on "Saved" tab (empty state) if none saved
6. "Clear Filter" link in empty state tapped → resets to "All"

### Edge Cases
- **Category not in data**: If no places have `category: "Nightlife"`, the "Nightlife" chip is not rendered
- **All places same category**: Only "All" and that category chip render; tapping the category chip shows all places (same result as "All")
- **Case mismatch in data**: `place.category = "food"` vs chip label "Food" → case-insensitive comparison handles correctly
- **Saved tab + category filter**: Only saved places of the selected category shown; if none, "No saved [Category] places yet"
- **Long category name** (hypothetical): Chip truncates gracefully with ellipsis or wraps onto second line within chip (should not break the row layout)
- **Rapid chip tapping**: Tapping multiple chips quickly results in the last tapped chip being active; no race conditions (synchronous state update)
