## User Story

**As a** guest,
**I want** to make a payment against my outstanding bill directly through the app,
**So that** I can settle my account at any time without visiting the front desk.

---

## Background / Context

In-stay payment capability is a key differentiator for Destinity vButler within the Browns Hotels & Resorts digital strategy. Guests, particularly business travellers, value the ability to settle partial or full balances on their own schedule. Phase 1 is a fully simulated payment flow — no payment gateway is integrated — but the UX must be indistinguishable from a real payment interaction to set accurate expectations for Phase 2 live integration. All card details entered are discarded immediately and never persisted.

## Acceptance Criteria

- [ ] The payment screen is accessible at `#/payment`, navigated to from the Billing Hub's "Pay Now" button (and from US-14.5 for advance payments)
- [ ] The current **outstanding balance** is displayed prominently at the top of the screen
- [ ] **Payment amount options** are presented:
  - "Full Balance" radio option (default): pre-fills the amount with the total outstanding balance
  - "Partial Amount" radio option: reveals a numeric input field; the guest may enter any amount between LKR 1 and the outstanding balance; entering an amount exceeding the balance shows a validation error
- [ ] **Payment method radio cards** are displayed with icon + label:
  - Credit / Debit Card
  - Digital Wallet (e.g. PayHere, FriMi — shown as a placeholder for Phase 2)
- [ ] When "Credit / Debit Card" is selected, a **card details form** is displayed:
  - Card number: text input, auto-formats as `#### #### #### ####` as the guest types (masking applied); last 4 digits remain visible
  - Expiry date: `MM/YY` format, auto-formatted
  - CVV: 3–4 digit input, shown as `•••` (masked/type="password")
  - Cardholder name: text input
  - None of these values are written to localStorage at any point
- [ ] When "Digital Wallet" is selected, the card fields are hidden and a placeholder message is shown: "Digital wallet payments coming soon — please use Card for now"
- [ ] A **"Confirm Payment" primary button** (full-width, brand colour) is displayed at the bottom of the form
- [ ] Tapping "Confirm Payment":
  1. Shows a **2-second processing overlay** with a spinner and text "Processing your payment…"
  2. On completion (after 2 seconds): transitions to the **success state**
- [ ] The **success state** displays:
  - Amount paid (LKR formatted)
  - Last 4 digits of the card number (captured from input, not stored beyond this display)
  - A unique transaction reference (format: `PAY-YYYYMMDD-XXXX`)
  - Timestamp of payment (current date/time, formatted as "15 Mar 2026, 02:30 PM")
  - "Your payment is confirmed" heading with a green checkmark
- [ ] On payment success:
  - A payment entry is appended to `vb:reservations[n].folio[]` as: `{ id, date: today, description: "Card Payment — Visa ×[last4]", quantity: 1, unitPrice: amountPaid, total: -amountPaid, type: "payment", category: "Payments", transactionRef, paidAt }`
  - The outstanding balance in `vb:reservations[n]` is updated accordingly (recalculated from folio)
  - An in-app notification is created: "Payment of LKR X,XXX received — Thank you!"
- [ ] A **security disclaimer** is displayed directly below the card form (always visible): "Your payment is processed securely. Card details are not stored."
- [ ] All card input fields are cleared from the DOM immediately after the "Confirm Payment" button is tapped (before the success state renders)

## UI / UX Notes

- Page route: `#/payment`
- Page header: "Make a Payment" with a `payments` icon; back chevron navigates to `#/billing`
- **Outstanding balance display**: large card at top; balance in 28px bold brand colour; label "Outstanding Balance" in 12px muted above
- **Payment amount section**: two radio cards side by side (or stacked on narrow mobile); "Full Balance" shows the amount; "Partial Amount" reveals a numeric input with "LKR" prefix label
- **Payment method cards**: icon + label in a card row; border highlights on selection (brand colour border); card icon: `credit_card`; wallet icon: `account_balance_wallet`
- **Card form layout**: stacked inputs, full-width; card number field is widest; expiry + CVV on same row (50%/50%); cardholder name full-width; all inputs use `border-radius: 8px`, subtle shadow on focus
- **Security disclaimer**: 12px muted text, padlock icon prefix (`lock`), centred below the form
- **Processing overlay**: full-screen semi-transparent overlay (`rgba(0,0,0,0.6)`); white circular spinner (CSS animation); "Processing your payment…" in white 16px; cannot be dismissed by guest
- **Success state**: replaces page content; large green checkmark (animated scale-in); "Payment Confirmed" in Playfair Display 22px; amount + card last 4 + transaction ref + timestamp in structured card; two buttons: "View My Bill" (navigates to `#/billing`) + "Return to Home" (navigates to `#/`)
- Dark mode: all form elements, overlay, success state use dark surface variables

## Technical Notes

- **Card number formatting** (Alpine.js input handler):
  ```js
  formatCardNumber(value) {
    return value.replace(/\D/g, '').replace(/(.{4})/g, '$1 ').trim().slice(0, 19);
  }
  ```
  Bind with `x-on:input="cardNumber = formatCardNumber($event.target.value)"`
- **CVV masking**: `<input type="password" maxlength="4" autocomplete="off">`
- **Expiry formatting**: `value.replace(/\D/g,'').replace(/^(\d{2})(\d)/, '$1/$2').slice(0,5)`
- **Payment simulation**: `setTimeout(() => { this.processPaymentSuccess(); }, 2000)` triggered on Confirm tap
- **Last 4 extraction**: `this.cardNumber.replace(/\s/g,'').slice(-4)` — captured into a local variable before clearing the input
- **Clear card fields**: set `this.cardNumber = ''; this.expiry = ''; this.cvv = ''; this.cardholderName = ''` immediately when Confirm is tapped (before `setTimeout`)
- **Folio write**:
  ```js
  const payment = {
    id: generateId(),
    date: dayjs().format('YYYY-MM-DD'),
    description: `Card Payment — Visa ×${last4}`,
    quantity: 1, unitPrice: amountPaid, total: -amountPaid,
    type: 'payment', category: 'Payments',
    transactionRef: `PAY-${dayjs().format('YYYYMMDD')}-${Math.floor(1000+Math.random()*9000)}`,
    paidAt: dayjs().toISOString()
  };
  ```
- **Notification**: append to `vb:notifications[]`: `{ id, type: 'payment_confirmed', message: 'Payment of LKR X,XXX received — Thank you!', createdAt, read: false }`
- **Alpine.js** `x-data="paymentForm()"`: `outstandingBalance`, `amountMode` ('full'|'partial'), `partialAmount`, `paymentMethod` ('card'|'wallet'), `cardNumber`, `expiry`, `cvv`, `cardholderName`, `processing`, `success`, `transactionRef`, `last4`
- Form validation before Confirm: card number 16 digits, expiry valid MM/YY format, CVV 3-4 digits, cardholder name not empty; show inline error messages if any fail

## Test Cases

### Happy Path
1. Guest taps "Pay Now" on Billing Hub → navigates to `#/payment`; outstanding balance "LKR 45,250" displayed
2. "Full Balance" selected by default; amount = LKR 45,250
3. Guest selects "Credit / Debit Card"; card form appears
4. Enters card: "4111 1111 1111 1111" (auto-formatted), expiry "12/26", CVV "123", name "Kavindi Perera"
5. Security disclaimer visible below form
6. Taps "Confirm Payment" → processing overlay appears for 2 seconds
7. Success state: "Payment Confirmed", "LKR 45,250 paid", "Visa ×1111", "PAY-20260315-XXXX", timestamp
8. `vb:reservations[n].folio[]` has new payment entry with `total: -45250`
9. Notification created: "Payment of LKR 45,250 received — Thank you!"
10. Guest taps "View My Bill" → Billing Hub shows balance = 0, "Paid in Full"

### Edge Cases
- **Partial amount exceeds balance**: Input shows validation error; Confirm button disabled until corrected
- **Partial amount = 0 or negative**: Validation error "Please enter a valid amount"
- **Digital Wallet selected**: Card form hidden; "coming soon" message; Confirm button still present (for Phase 2 readiness) but should show appropriate handling
- **Card number < 16 digits**: Inline validation error on Confirm tap; form not submitted
- **Expired card date (MM/YY in past)**: Validation error on expiry field
- **Pop-up / autofill**: `autocomplete="off"` on CVV; browser autofill does not accidentally save card details
- **Double-tap Confirm**: Button disabled after first tap; second tap ignored; no duplicate folio entries
- **Navigate away during processing**: If guest presses back during the 2-second overlay, processing completes (payment logged) but success screen is not shown — acceptable Phase 1 behaviour; consider cancellation in Phase 2
