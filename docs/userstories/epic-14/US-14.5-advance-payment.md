## User Story

**As a** guest with an upcoming reservation,
**I want** to pay a deposit or full balance in advance through the app,
**So that** I can complete financial obligations before arriving and enjoy a smoother check-in experience.

---

## Background / Context

Pre-arrival payments reduce check-in friction and provide Browns Hotels & Resorts with financial security ahead of the guest's arrival. For guests completing the pre-arrival digital checklist, advance payment is a natural step in the preparation journey. This story reuses the payment flow from US-14.4 with contextual adjustments for the pre-arrival scenario — reinforcing the frictionless digital check-in vision of the Destinity vButler platform.

---

## Acceptance Criteria

- [ ] The Advance Payment flow is accessible from two entry points:
  1. **Pre-Arrival Checklist** (upcoming reservation) → "Advance Payment" checklist item → tapping navigates to `#/payment/advance/{resvId}`
  2. **Reservation Detail** page for an **upcoming** reservation where `reservation.balance > 0` → "Pay Now" button
- [ ] The payment screen displays a **context banner** at the top identifying this as a pre-arrival payment: "Advance Payment — [Property Name], arriving [Check-in Date formatted as 'D MMM YYYY']" in brand colour with a calendar icon
- [ ] The **outstanding balance** for the upcoming reservation is displayed; it is sourced from `vb:reservations[n].folio[]` for the specific reservation (not the active stay)
- [ ] **Payment options** presented:
  - **Deposit**: shown if `reservation.depositRequired` is configured and the deposit has not already been paid; amount = `reservation.depositAmount` (LKR); label = "Pay Deposit — LKR X,XXX"
  - **Full Balance**: always shown; amount = total outstanding balance; label = "Pay Full Balance — LKR X,XXX"
  - If only Full Balance is applicable (no deposit configured or deposit already paid), only "Full Balance" is shown without a radio choice
- [ ] The payment method and card fields are identical to US-14.4 (Credit/Debit Card | Digital Wallet with same card form and security disclaimer)
- [ ] On **payment success**:
  - The outstanding balance in `vb:reservations[n]` is updated (payment folio entry appended with negative total)
  - If the payment was a **Deposit** payment and a pre-arrival checklist item `"Advance Payment"` exists, its `completed` field is set to `true`
  - A success state is shown: "Advance Payment Confirmed", amount, transaction reference, property name and arriving date
  - An in-app notification is created: "Advance payment of LKR X,XXX confirmed for [Property Name] — arriving [Date]"
- [ ] If the reservation's balance is already fully paid before the guest opens this screen, the screen shows: "Your balance is settled — nothing to pay" with a green checkmark; both entry point buttons are hidden or disabled when balance = 0

## UI / UX Notes

- Page route: `#/payment/advance/{resvId}`
- Page header: "Advance Payment" with back chevron; brand top bar
- **Context banner**: full-width; brand-tinted background (light brand primary, 10% opacity); brand-coloured icon and text; 14px; border-bottom in brand accent; padding 12px 16px; appears immediately below the page header

- Dark mode: context banner tint adjusts to dark surface variant

## Technical Notes

- Route parsing: SPA router extracts `resvId` from `#/payment/advance/{resvId}`; component looks up `vb:reservations` to find matching reservation by id
- **Reservation lookup**:
  ```js
  const reservations = JSON.parse(localStorage.getItem('vb:reservations') || '[]');
  this.reservation = reservations.find(r => r.id === resvId);
  ```
- **Balance calculation**: same as US-14.1 — sum charges minus sum payments from `reservation.folio[]`
- **Deposit logic**:
  ```js
  this.showDeposit = this.reservation.depositRequired &&
    this.reservation.depositAmount > 0 &&
    !this.reservation.folio.some(f => f.type === 'payment' && f.description.includes('Deposit'));
  ```
- **Checklist update on deposit payment**:
  ```js
  const checklist = JSON.parse(localStorage.getItem(`vb:checklist:${resvId}`) || '[]');
  const item = checklist.find(i => i.key === 'advance_payment');
  if (item) { item.completed = true; item.completedAt = dayjs().toISOString(); }
  localStorage.setItem(`vb:checklist:${resvId}`, JSON.stringify(checklist));
  ```
- **Folio write**: identical to US-14.4 payment entry; `description` differentiates: "Advance Deposit — Visa ×[last4]" or "Advance Payment — Visa ×[last4]"
- **Alpine.js** `x-data="advancePayment(resvId)"` extends US-14.4 logic; additional properties: `reservation`, `reservationId`, `showDeposit`, `depositAmount`, `paymentMode` ('deposit' | 'full')
- Notification: `{ message: 'Advance payment of LKR X,XXX confirmed for [Property] — arriving [Date]', type: 'advance_payment_confirmed' }`

## Test Cases

### Happy Path
1. Guest opens Pre-Arrival Checklist for upcoming reservation; "Advance Payment" item is incomplete
2. Taps "Advance Payment" → navigates to `#/payment/advance/{resvId}`
3. Context banner: "Advance Payment for Browns Boutique Hotel, arriving 20 Apr 2026"
4. Two payment options: "Pay Deposit — LKR 15,000" and "Pay Full Balance — LKR 75,500"
5. Guest selects "Pay Deposit"; card form appears
6. Enters card details; taps "Confirm Payment"
7. Processing overlay for 2 seconds
8. Success state: "Advance Payment Confirmed — LKR 15,000", transaction ref, property + arriving date
9. `vb:reservations[n].folio[]` has deposit payment entry
10. Pre-Arrival Checklist "Advance Payment" item is now `completed: true`
11. In-app notification created

### Edge Cases
- **No deposit configured**: Only "Full Balance" option shown; no radio choice rendered; guest can pay full balance directly
- **Balance already = 0**: "Nothing to pay" state shown; entry point buttons in Checklist and Reservation Detail are hidden/disabled
- **Deposit already paid**: Deposit option hidden; only "Full Balance" shown; balance reflects remaining amount after deposit
- **Full balance payment**: Entire outstanding balance settled; checklist item completed; "View Reservation" link available in success state
- **Invalid resvId in URL**: Reservation not found → show "Reservation not found" error with link to home
- **Upcoming reservation becomes active** (check-in day): Advance payment route still works; context banner updates to reflect same-day arrival if needed
- **Double-tap Confirm**: Button disabled after first tap; duplicate payment entry prevented
