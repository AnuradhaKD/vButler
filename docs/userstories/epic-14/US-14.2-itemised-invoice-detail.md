## User Story

**As a** guest,
**I want** to see a full line-by-line breakdown of every charge and payment on my account,
**So that** I can verify each item and understand exactly what I am being billed for.

---

## Background / Context

Itemised billing is a legal and trust requirement in the hospitality industry. Guests expect to review every charge before check-out, and discrepancies discovered at the desk create friction and negative sentiment. By surfacing this detail digitally through Destinity vButler, Browns Hotels & Resorts empowers guests to review at their leisure — reducing check-out queues and dispute frequency. The day-grouped layout with tax breakdowns mirrors a formal hotel invoice.

## Acceptance Criteria

- [ ] The itemised invoice detail is displayed below the Charge Summary on the Billing Hub (`#/billing`) or on a dedicated screen (`#/billing/detail`); accessible via "View Full Invoice" link on the hub
- [ ] Each folio line item renders: date (formatted as "5 Apr"), description, quantity (shown only if `quantity > 1`, e.g. "× 2"), unit price (if quantity > 1), and line total in LKR
- [ ] Charge line items are **grouped by day** with a **day header row** showing the date (e.g. "Wednesday, 5 April") and a **day subtotal row** at the end of each day's group (sum of all charges for that day, excluding payments)
- [ ] A **tax breakdown row** is rendered per day group if `chain.json` has `taxRate` configured (e.g. "Service Charge & Taxes — LKR X,XXX"); tax = day subtotal × taxRate; the tax row is visually subordinate (smaller, muted)
- [ ] **Payment entries** (folio items where `type: "payment"`) are displayed with a **green** text colour and a **negative sign** prefix (e.g. "– LKR 20,000") to clearly distinguish them as deductions from the running total
- [ ] A **running total** is displayed at the very bottom of the full list: "Total Outstanding: LKR X,XXX" or "Paid in Full" if balance is zero; this total includes all charges, taxes, and payments
- [ ] **Horizontal rules** (or equivalent visual separators — light grey lines) separate each day group from the next
- [ ] A **"Download Invoice" button** is displayed at the top-right of the page/section header; tapping it triggers the PDF download flow (US-14.3)
- [ ] The itemised view respects the date range filter and search query set on the Billing Hub — it shows only filtered/searched entries

## UI / UX Notes

- Section/page header: "Invoice Detail" or "Full Invoice" with a back chevron; "Download Invoice" button top-right (icon: `download` or `picture_as_pdf`, brand text colour)
- **Day header row**: light grey background (`#f5f5f5` light / dark surface in dark mode); date text 13px uppercase bold brand colour; left-padded 12px; full width separator
- **Line item row**: 14px description; quantity "× N" in muted 12px (shown only when qty > 1); unit price in muted 12px if qty > 1; total right-aligned 14px; 12px vertical padding; left 16px padding
- **Day subtotal row**: slightly darker background than day header; "Subtotal" label left; total right-aligned; font-weight 600; 13px
- **Tax row**: muted 12px italic; "Service Charge & Taxes (10%)" or equivalent; right-aligned tax amount; same background as subtotal row
- **Payment rows**: description in 14px; amount in green bold "– LKR X,XXX" right-aligned; optionally a small payment icon prefix (`payments`)
- **Running total row**: bold 16px; brand colour amount; strong top border; padding 16px; "Total Outstanding" or "Paid in Full"
- Horizontal rule between day groups: `<hr>` styled as `border-top: 1px solid var(--border-color)`
- Empty state (no charges or all filtered out): "No items to display" with a filter/clear hint
- Long descriptions wrap to second line; totals remain right-aligned regardless of description length (use CSS flexbox: description `flex: 1`, total `flex-shrink: 0`)
- Dark mode: day header, subtotal row, tax row backgrounds adjusted to dark surface variants; payment green remains accessible contrast

## Technical Notes

- **Day grouping logic**:
  ```js
  groupByDay(entries) {
    return entries.reduce((groups, entry) => {
      const day = dayjs(entry.date).format('YYYY-MM-DD');
      if (!groups[day]) groups[day] = [];
      groups[day].push(entry);
      return groups;
    }, {});
  }
  ```
  - Render using `Object.keys(groups).sort()` to ensure chronological order
- **Day subtotal**: `group.filter(e => e.type !== 'payment').reduce((s, e) => s + e.total, 0)`
- **Tax calculation**: `daySubtotal * (chain.taxRate || 0)` — `chain.taxRate` loaded from `chain.json` (e.g. `0.10` for 10%); if `taxRate` is 0 or undefined, tax row is hidden
- **Running total** (includes tax):
  ```js
  const totalCharges = folio.filter(e => e.type !== 'payment').reduce((s, e) => s + e.total, 0);
  const totalTax = /* sum of per-day taxes */;
  const totalPaid = folio.filter(e => e.type === 'payment').reduce((s, e) => s + Math.abs(e.total), 0);
  const outstanding = totalCharges + totalTax - totalPaid;
  ```
- **Alpine.js** component (`x-data="invoiceDetail()"` or part of `billingHub()`):
  - `groupedEntries` — getter: calls `groupByDay(filteredFolio)` (using the filtered folio from US-14.1)
  - `daySubtotal(entries)`, `dayTax(entries)` — methods
  - `runningTotal` — computed getter
- Payment row rendering: `x-bind:class="{ 'text-green': entry.type === 'payment' }"`; format: `entry.type === 'payment' ? '– LKR ' + format(Math.abs(entry.total)) : 'LKR ' + format(entry.total)`
- "Download Invoice" button: `@click="generatePDF()"` — PDF generation handled by US-14.3 component; can be imported as a method or triggered via event

## Test Cases

### Happy Path
1. Guest navigates to Billing Hub → itemised detail visible below summary (or taps "View Full Invoice")
2. Day 1 group: header "Tuesday, 3 April", 2 charge rows, subtotal row, tax row (10%)
3. Day 2 group: header "Wednesday, 4 April", 3 charge rows, 1 payment row (green, "– LKR 20,000"), subtotal, tax
4. Payment row: green text, negative sign, description "Card Payment — Visa ×4321"
5. Running total at bottom: "Total Outstanding: LKR 25,750"
6. "Download Invoice" button visible top-right → tapping triggers PDF generation

### Edge Cases
- **Single day with no charges**: Day header does not render; no empty groups
- **All paid (balance = 0)**: Running total shows "Paid in Full" in green; all payment rows visible
- **`taxRate` = 0 or missing**: Tax rows hidden for all day groups
- **Quantity > 1 items**: Line shows "× 3" and unit price; total = qty × unit; formatting verified
- **Quantity = 1**: Quantity and unit price columns hidden; only description and total shown
- **Date filter applied**: Only entries within filter range grouped and shown; day headers outside range do not appear
- **Search applied**: Only matching entries shown; day groups with all entries filtered out are hidden (no empty day headers)
- **Very large folio (50+ items)**: Page scrolls smoothly; day groups render without layout issues
